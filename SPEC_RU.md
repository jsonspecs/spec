# Спецификация поведения JSONSpecs

**Версия:** 1.0.0 · **Статус:** Release Candidate — тег `v1.0.0` вешается после
кросс-реализационного сравнения на живом стенде (см. README репозитория, раздел
о релизном процессе). English version: [SPEC.md](SPEC.md) — при расхождениях
нормативен английский текст.

Эта спецификация — канон поведения рантайма для данной версии. Она описывает только
ожидаемое поведение. Аргументация каждого решения живёт в отдельном документе —
`DECISIONS_RU.md` / `DECISIONS.md`, — который, в свою очередь, ссылается на
прикладной прототип (`source/`), на опыте которого эти решения приняты.

**Нотация.** `[D1]`…`[D14]` — ссылки на нумерованные решения в `DECISIONS_RU.md`;
`[DR-I]`, `[DR-II]`, `[DR-III]` — на его аддендумы по частям I–III драфт-ревью.
MUST / MUST NOT / SHOULD / MAY — по RFC 2119 (в переводе: ОБЯЗАН / НЕ ДОЛЖЕН /
СЛЕДУЕТ / МОЖЕТ; ниже используются английские ключевые слова).

---

## 1. Введение и скоуп

Спецификация определяет наблюдаемое поведение рантайма правил JSONSpecs:

1. формат артефактов правил и снэпшотов;
2. семантику встроенных операторов — что они решают, а не как;
3. формат и содержание результата исполнения.

Две конформные реализации, получив один и тот же снэпшот, один `payload` и один
`context`, MUST выдать идентичные нормативные результаты (нормативная поверхность
определена в разделе 7) и MUST принимать/отвергать одно и то же множество входов.

Спецификация не определяет API, имена функций, внутренние представления, стратегию
компиляции, момент валидации, характеристики производительности и транспортные
лимиты на размер payload или результата. Реализация MAY компилировать снэпшоты
заранее, интерпретировать напрямую, строить любые промежуточные структуры или не
строить никаких — эти выборы невидимы для конформанса. `[D12]`

Сквозной принцип проектирования: **DSL никогда не подразумевает решений за
аналитика.** Обязательно ли поле, обязан ли присутствовать контекст, какой код и
текст даёт ошибка отсутствия — каждое из этого есть явное правило, написанное
аналитиком, и никогда — побочный эффект value-оператора или удобство рантайма
(см. D13, D14).

Кастомные операторы — вне спецификации. Спецификация определяет только интерфейс
расширения (форму результата оператора и правило отвержения неизвестного оператора —
раздел 4); поведение любого не-встроенного оператора — обещание конкретной
реализации, а не спецификации. `[D10]`

## 2. Модель данных

### 2.1 Значения

*Значение* — это JSON-значение: `null`, boolean, число, строка, массив или объект.
`undefined` в модели данных не существует. Там, где хост-язык различает
«отсутствует» и «null», нормативно только JSON-видимое различие: ключа нет — против
ключа со значением `null`.

Все входы (артефакты, снэпшот, payload, context) и нормативный результат —
JSON-документы. Структуры, не проходящие через JSON-сериализацию (циклические
ссылки, нефинитные числа, хост-специфичные типы), — вне модели; граница рантайма
MUST их отвергнуть (раздел 5, валидация входа).

**Максимальная глубина.** Артефакты, payload, context и нормализованный результат
имеют максимальную глубину вложенности JSON — **256**. Входы, превышающие её,
отвергаются (раздел 5). `[D9]`

**Зарезервированные ключи.** Ключи объектов `__proto__`, `prototype` и
`constructor` запрещены в payload и context на любой глубине. Это правило контракта,
а не платформенная защита: реализации на платформах без прототипной семантики MUST
отвергать эти ключи точно так же — иначе одинаковые входы давали бы разные вердикты
на разных реализациях. `[D9]`

### 2.2 Числа

Числа интерпретируются как **IEEE 754 binary64**. `[D1]`

- Целые в диапазоне ±(2^53 − 1) представляются точно.
- JSON-число, математическое значение которого непредставимо в binary64 точно
  (целое вне безопасного диапазона или десятичная дробь вроде `0.1` там, где её
  binary64-округление наблюдаемо), — *вне гарантии детерминизма*: реализации MUST
  применять округление round-to-nearest-even, и IEEE-754-конформные платформы на
  практике совпадут, но авторам правил SHOULD NOT полагаться на сравнения на краю
  точности binary64.
- `1` и `1.0` — одно значение. `is_integer` принимает любое число с нулевой дробной
  частью: `1.0` проходит, `1.5` — нет.

**Числовые строки.** В упорядоченных сравнениях (§2.5) строка *числовая* тогда и
только тогда, когда соответствует грамматике:

```
^[+-]?[0-9]+(\.[0-9]+)?([eE][+-]?[0-9]+)?$
```

Числовая строка конвертируется в binary64 с округлением round-to-nearest-even.
Строки вне грамматики числами не являются, что бы ни принимал снисходительный
парсер хост-языка (`" 5"`, `"0x10"`, `"5,0"`, `"Infinity"` — не числа).

### 2.3 Строки

Строки — последовательности кодовых точек Unicode.

**Длина.** Везде, где спецификация измеряет длину строки (`length_equals`,
`length_max`), единица — **кодовая точка Unicode**. `[D2]` Суррогатная пара
считается за одну. Реализации, у которых нативная длина строки — UTF-16 code units
(JavaScript, Java), MUST считать кодовые точки явно.

Графемные кластеры не используются никогда: их определение зависит от версии
Unicode и сделало бы результаты зависимыми от релиза.

### 2.4 Равенство

Два значения *равны*, когда:

- оба `null`; или
- оба booleans с одинаковым значением; или
- оба числа с одним binary64-значением (`1 == 1.0`); или
- обе строки с одинаковой последовательностью кодовых точек.

Значения разных типов не равны никогда: `1` ≠ `"1"`, `false` ≠ `0`, `null` ≠ `""`.
Коэрции нет. Массивы и объекты в этом отношении не равны ничему (равенство
определено для скалярного сравнения в операторах; глубокого структурного равенства
в операторной модели нет).

### 2.5 Упорядоченное сравнение

Упорядоченное сравнение (`greater_than`, `less_than`, `field_*_field`) работает над
двумя операндами, классифицируемыми независимо:

| Операнд | Классификация |
| --- | --- |
| JSON-число | **число** |
| строка по числовой грамматике (§2.2) | **число** (конвертируется в binary64) |
| строка по `^[0-9]{4}-[0-9]{2}-[0-9]{2}$`, являющаяся валидной датой пролептического григорианского календаря | **дата** |
| всё остальное (booleans, `null`, массивы, объекты, прочие строки) | **неклассифицируемо** |

Календарная валидность включает диапазон месяцев 01–12, валидный для месяца
диапазон дней и правила високосных лет для февраля; `2026-02-30` и `2026-13-01` —
не даты. Диапазон лет — 0000–9999. Порядок дат — хронологический (эквивалентно:
лексикографический над канонической строкой `YYYY-MM-DD`).

Сравнение *определено* только когда оба операнда классифицируются в **один** вид
(оба числа или обе даты). Если любой операнд неклассифицируем или виды различаются,
сравнение *неопределено*, и оператор фейлится (check → FAIL, предикат → UNDEFINED).

Классификация дизъюнктна по построению: никакая строка не является одновременно
числовой и датой.

### 2.6 Пустота

Поле *пусто*, когда оно:

- **отсутствует** — его путь не резолвится в плоском payload (§2.7); или
- `null`; или
- `""` (пустая строка).

`false`, `0`, `[]` и `{}` — **не** пусты.

### 2.7 Пути, флаттенинг и резолюция полей

**Синтаксис пути.** Ссылка на поле — dot-notation-путь по плоской карте payload:

```
order.amount
accounts[0].balance
accounts[*].balance          ← wildcard, §3.6
$context.currentDate         ← доступ к контексту
```

**Флаттенинг.** Рантайм принимает payload либо вложенным JSON-объектом, либо уже
плоской картой путь → значение. Флаттенинг определён нормативно и идемпотентен
(плоский вход проходит без изменений):

- Ключ `k` вложенного объекта под префиксом `p` порождает пути `p.k…`; элемент
  массива с индексом `i` под префиксом `p` — пути `p[i]…`.
- **Листья** — это: скаляры (`null`, boolean, число, строка), пустой объект `{}` и
  пустой массив `[]`. Путь листа отображается в его значение.
- **Непустой контейнер сам в плоской карте не присутствует** — только пути его
  потомков. Для `{"a": [1]}` плоская карта — `{"a[0]": 1}`; путь `a` *отсутствует*.
  Для `{"a": []}` плоская карта — `{"a": []}`, и `a` резолвится в значение `[]`
  (непустое по §2.6).

**Резолюция.** Резолюция пути `f` по плоской карте даёт либо
*(присутствует, значение)*, либо *отсутствует*. Частичной резолюции, фолбэков и
значений по умолчанию нет.

**Доступ к контексту.** Пути с зарезервированным префиксом `$context.` резолвятся
по входу `context` вместо payload, с той же семантикой резолюции. Wildcards внутри
путей `$context.*` не допускаются (отвержение артефакта). Префикс зарезервирован:
ключ payload, буквально названный `$context`, не имеет специального смысла и
недостижим через пути `$context.*`.

Legacy-механизма передачи контекста внутри payload под ключом `__context` в этой
спецификации **не существует**: `__context` — обычный ключ payload без специальной
семантики. `[D11]`

## 3. Семантика операторов

### 3.1 Модель оператора

*Правило* применяет один оператор к payload (и, возможно, контексту). Роли
операторов:

- **check**-операторы возвращают `OK` или `FAIL`. `FAIL` порождает issue (раздел 5).
- **predicate**-операторы возвращают `TRUE`, `FALSE` или `UNDEFINED`. Предикаты не
  порождают issues; они питают `when`-выражения и предикатную агрегацию. Перед
  комбинированием листовых результатов в `when`-выражении `UNDEFINED`
  преобразуется в `FALSE`.

Поведение оператора — чистая функция от: разрезолвленных значений полей, параметров
правила (`value`, `value_field`, `fields`, `dictionary`, `flags`), содержимого
словаря по ссылке и — для `$context.*` — контекста. Операторы MUST NOT зависеть от
порядка исполнения, часов, локали и любого иного ambient-состояния.

**Поведение при отсутствии поля — два класса операторов.** `[D13]`

- **Presence-семантика** (`not_empty`, `is_empty`, `not_true`, `any_filled`):
  отсутствие — часть их предметной области; поведение на отсутствующем поле
  определено индивидуально в таблице ниже.
- **Value-семантика** (все остальные): они ограничивают значение, *если оно есть*.
  При отсутствии поля оператор **не вызывается**: check даёт OK и не порождает
  issue (*skip*); предикат даёт UNDEFINED. Реализации SHOULD записать skip как
  событие trace (информативная поверхность; нормативный результат не затронут).
  Обязательность всегда выражается отдельным presence-правилом
  (`not_empty` / `any_filled`) — и никогда не подразумевается value-проверкой. Это
  сознательный принцип проектирования DSL, а не равнение на отрасль: обязательность
  поля — явное решение аналитика, и ошибка *отсутствия* — отдельная диагностика от
  ошибки *оператора*: она заслуживает собственного правила со своими `code` и
  `message`, написанными аналитиком, а не generic-ошибки, вшитой в каждый
  value-оператор. (Совпадение с разделением `pattern`/`enum` и `required` в
  JSON Schema — побочное, не мотивирующее.)

Для операторов `field_*_field` правило value-семантично по обоим операндам: если
отсутствует путь `field` или путь `value_field`, проверка скипается.

**Строго строковые операторы** `[D3][DR-I]`: операторы `contains`, `matches_regex`
и `not_matches_regex` требуют строкового значения поля; любое нестроковое значение
(включая числа) — `FAIL` / `FALSE`. Хостовая стрингификация не происходит никогда.
Замечание для миграции: правила, матчившие числовые поля payload (например,
интегратор шлёт ИНН JSON-числом), меняют вердикт с прохода на FAIL — это вскрытие
ошибки типизации payload, а не её маскировка.

### 3.2 Check-операторы

Встроенный набор check-операторов. Колонка «Отсутствие» — поведение, когда поле не
резолвится (§2.7): presence-операторы определяют его индивидуально;
value-операторы показывают **skip** (OK, без issue, без вызова — §3.1). Все
сравнения — по равенству §2.4 и упорядоченному сравнению §2.5; коэрции сверх
классификации числовых строк и дат из §2.5 не бывает.

| Оператор | Параметры | OK когда | Отсутствие |
| --- | --- | --- | --- |
| `not_empty` | — | поле не пусто (§2.6) | FAIL |
| `is_empty` | — | поле пусто (§2.6) | OK |
| `not_true` | — | значение — что угодно, кроме boolean `true` | OK |
| `any_filled` | `fields: path[]` | хотя бы одно из перечисленных полей не пусто (§2.6); `field` не используется | отсутствие = пусто; FAIL, когда все перечисленные поля пусты |
| `is_boolean` | — | значение — boolean | skip |
| `is_string` | — | значение — строка | skip |
| `is_number` | — | значение — число | skip |
| `is_integer` | — | значение — число с нулевой дробной частью | skip |
| `equals` | `value` | значение равно `value` (§2.4) | skip |
| `not_equals` | `value` | значение не равно `value` (§2.4) | skip |
| `contains` | `value: string` | значение — строка, содержащая `value` как подстроку (вхождение последовательности кодовых точек; пустой `value` содержится в любой строке) | skip |
| `matches_regex` | `value: pattern`, `flags?` | значение — строка, содержащая совпадение паттерна (§3.4; семантика поиска — незаякоренная, если паттерн не якорится сам) | skip |
| `not_matches_regex` | `value: pattern`, `flags?` | значение — строка, **не** содержащая совпадений паттерна | skip |
| `greater_than` | `value: число \| дата-строка` | сравнение определено (§2.5) и поле > value | skip |
| `less_than` | `value: число \| дата-строка` | определено и поле < value | skip |
| `length_equals` | `value: число` | значение — строка ровно из `value` кодовых точек (§2.3); нестрока → FAIL `[D2][D3]` | skip |
| `length_max` | `value: число` | значение — строка не более чем из `value` кодовых точек; нестрока → FAIL `[D2][D3]` | skip |
| `field_equals_field` | `value_field: path` | оба поля присутствуют и равны (§2.4) | skip, если любое отсутствует |
| `field_not_equals_field` | `value_field: path` | оба присутствуют и не равны | skip, если любое отсутствует |
| `field_greater_than_field` | `value_field: path` | оба присутствуют, сравнение определено, поле > второго; неопределено → FAIL | skip, если любое отсутствует |
| `field_less_than_field` | `value_field: path` | оба присутствуют, определено, поле < второго | skip, если любое отсутствует |
| `field_greater_or_equal_than_field` | `value_field: path` | оба присутствуют, определено, поле ≥ второго | skip, если любое отсутствует |
| `field_less_or_equal_than_field` | `value_field: path` | оба присутствуют, определено, поле ≤ второго | skip, если любое отсутствует |
| `in_dictionary` | `dictionary: {type:"static", id}` | значение матчит запись словаря (§3.5) | skip |
| `not_in_dictionary` | `dictionary: {type:"static", id}` | значение не матчит **ни одну** запись словаря (§3.5) | skip |

Примечания.

- `any_filled` принимает `fields[]`; legacy-алиаса `paths[]` в спецификации не
  существует. `[D11]`
- `not_true`: `OK` для отсутствия, `null`, `""`, `false`, `0`, `"true"`, объектов —
  всего, кроме boolean `true`. FAIL — только строгий boolean `true`. Оператор
  presence-семантичен по замыслу (запрет, выполняющийся вакуумно).
- Для `greater_than`/`less_than` значение `value` в артефакте классифицируется по
  тем же правилам §2.5, что и операнд payload (JSON-число либо строка — числовая
  или валидная дата).
- Идиома «поле обязательно *и* должно удовлетворять X» — два правила: `not_empty` и
  value-проверка. Отсутствующее поле тогда даёт ровно один issue (про
  обязательность), а не каскад. `[D13]`

### 3.3 Predicate-операторы

Доступны как предикаты: `not_empty`, `is_empty`, `is_boolean`, `is_string`,
`is_number`, `is_integer`, `equals`, `not_equals`, `contains`, `matches_regex`,
`not_matches_regex`, `greater_than`, `less_than`, все шесть операторов
`field_*_field`, `in_dictionary`, `not_in_dictionary`.

**Недоступны как предикаты:** `any_filled`, `length_equals`, `length_max`,
`not_true`. Правило с `role: "predicate"` и одним из этих операторов отвергается.

Маппинг результата: где таблица check говорит OK, предикат возвращает `TRUE`; где
FAIL для *присутствующего* поля — `FALSE`. На **отсутствующем** поле:
value-семантичные предикаты возвращают `UNDEFINED` (§3.1); `is_empty` возвращает
`TRUE`; `not_empty` — `FALSE`. `UNDEFINED` преобразуется в `FALSE` на листе
`when`-выражения (раздел 5) и считается не-`TRUE` в предикатной агрегации (§3.6).

### 3.4 Регулярные выражения

`[D4]` Раздел определяет язык паттернов полностью. Это переносимое подмножество,
выбранное так, чтобы каждая мейнстримная бэкенд-платформа исполняла его штатным
regex-движком (строгое подмножество RE2, ECMAScript-с-`u`, `java.util.regex` и
non-backtracking-режима .NET). Паттерны подмножества исполняются за линейное время;
дополнительные ReDoS-эвристики не нужны и не определяются.

#### 3.4.1 Препроцессинг

Перед разбором к строке паттерна применяется ровно один проход замен: каждая пара
последовательных обратных слэшей `\\` заменяется одним `\`. Проход не итеративен.
`[D4.3]` (Следствие: на уровне JSON-исходника и `"^\\d+$"`, и `"^\\\\d+$"`
обозначают паттерн `^\d+$`.)

#### 3.4.2 Грамматика

После препроцессинга паттерн MUST соответствовать:

```ebnf
pattern      = alternation ;
alternation  = concat , { "|" , concat } ;
concat       = { element } ;
element      = anchor | quantified ;
anchor       = "^" | "$" ;
quantified   = atom , [ quantifier ] ;
quantifier   = ( "*" | "+" | "?" | counted ) ;            (* без lazy/possessive *)
counted      = "{" , int , [ "," , [ int ] ] , "}" ;      (* {n} {n,} {n,m}, n ≤ m *)
atom         = literal | "." | escape | class | group ;
group        = "(" , [ "?:" ] , alternation , ")" ;
class        = "[" , [ "^" ] , class-item , { class-item } , "]" ;
class-item   = class-atom , [ "-" , class-atom ]          (* диапазон: левый ≤ правый *)
             | class-escape ;
class-atom   = class-literal | escaped-meta | char-escape ;
escape       = escaped-meta | char-escape | class-escape ;
escaped-meta = "\" , ( "\" | "." | "*" | "+" | "?" | "(" | ")" | "[" | "]"
                     | "{" | "}" | "|" | "^" | "$" | "/" | "-" ) ;
char-escape  = "\n" | "\r" | "\t" ;
class-escape = "\d" | "\D" | "\w" | "\W" | "\s" | "\S" ;
int          = digit , { digit } ;                        (* значение ≤ 1000 *)
```

`literal` — любая кодовая точка, кроме метасимволов `\ . * + ? ( ) [ ] { } | ^ $`.
`class-literal` — любая кодовая точка, кроме `\ ] -` (литеральный `-` пишется
первым, последним или экранированным; литеральный `^` внутри класса — где угодно,
кроме первой позиции).

**Явно исключены** (их присутствие делает паттерн — и, следовательно, снэпшот —
невалидным): backreferences, lookahead/lookbehind `(?= (?! (?<= (?<!`, именованные
группы `(?<name>`, границы слова `\b \B`, инлайн-флаги `(?i)`, hex/unicode-эскейпы
`\xHH \uHHHH \x{…}` (символы Unicode пишутся литерально или экранируются на уровне
JSON-строки, которая уже даёт `\uXXXX`), POSIX-классы `[:alpha:]`, вложенные
классы, восьмеричные эскейпы и любой эскейп, не перечисленный в грамматике.

**Нормативные лимиты** (защита от расходящихся внутренних лимитов движков): каждый
`int` в квантификаторе MUST быть ≤ 1000; препроцессированный паттерн MUST быть
≤ 1024 кодовых точек. Паттерны, нарушающие грамматику или лимиты, делают артефакт
невалидным — это отвержение артефакта, а не рантайм-ошибка.

#### 3.4.3 Семантика матчинга

- Субъект — последовательность кодовых точек Unicode; `.` матчит любую одну
  кодовую точку, кроме U+000A LINE FEED. (JavaScript-реализации MUST компилировать
  с флагом `u` для code-point-семантики.)
- `\d` = `[0-9]`, `\w` = `[0-9A-Za-z_]`, `\s` = `[ \t\n\r\f\v]`, верхний регистр —
  дополнения. **ASCII-семантика независимо от дефолтов платформы** — движки с
  Unicode-осведомлёнными дефолтами (например, Rust `regex`) MUST компенсировать.
  `[D4.1]`
- Матчинг — **поиск**: паттерн совпадает, если совпадает любая подстрока (включая
  пустую). Авторы якорят `^`/`$` явно.
- Флаги: только `i`, `m`, `s`, каждый не более раза, в любом порядке; любая иная
  строка флагов делает артефакт невалидным.
  - `i` — регистронезависимость через **Unicode simple case folding**,
    culture-invariant (`и` матчит `И`; поведение турецкой локали `i`/`I` MUST NOT
    протекать). `[D4.2]`
  - `m` — `^`/`$` дополнительно матчат после/перед U+000A.
  - `s` — `.` матчит и U+000A.

### 3.5 Словари

Словарь — именованный список `entries`. Значение payload *матчит* запись, когда:

| Форма записи | Матчит когда |
| --- | --- |
| скаляр (строка, число, boolean; записи `null` делают артефакт невалидным) | значение равно записи (§2.4) |
| объект с полями `code` и/или `value` | значение равно `entry.code` **или** равно `entry.value` (участвуют все присутствующие поля) |
| объект без обоих полей | не матчит никогда (артефакт SHOULD отвергаться схемой; при наличии — не-матч) |

Матчинг — равенство §2.4: строгое, без коэрции; тип значения payload должен
совпадать с типом записи. `in_dictionary` — OK/TRUE, когда матчит любая запись;
`not_in_dictionary` — OK/TRUE, когда не матчит ни одна. `dictionary.type` MUST
быть `"static"`.

### 3.6 Wildcards и агрегация

#### 3.6.1 Резолюция wildcard и порядок перечисления `[D5]`

Путь `field` может содержать сегменты `[*]`. Сегмент `[*]` матчит ровно сегменты —
**неотрицательные целые** индексы в этой позиции плоской карты; ничего иного.
Резолюция даёт список конкретных путей, присутствующих в плоской карте.

Порядок перечисления нормативен:

1. индексы упорядочиваются **как числа, по возрастанию** (`a[2]` раньше `a[10]`;
   порядок вставки в плоскую карту не значит ничего; дыры допустимы —
   `a[0], a[2], a[5]` перечисляются в этом порядке);
2. при нескольких сегментах `[*]` кортежи индексов упорядочиваются
   лексикографически, **левый сегмент старше** («одометр»):
   `[0][0], [0][1], [1][0], [1][2], …`

Этот порядок определяет порядок поэлементных issues в результате и порядок
элементов, наблюдаемый агрегацией.

#### 3.6.2 Агрегация

Без объекта `aggregate` действуют дефолты: check → `EACH`, предикат → `ANY`.

Поля `aggregate`: `mode`, `onEmpty`, `summaryIssue` (только check + `ALL`), `op` и
`value` (только `COUNT`; `op` ∈ `== != > >= < <=`, дефолт `>=`; `value`
обязателен).

**Check-режимы.** Базовый оператор применяется к каждому элементу; элемент
*проходит*, когда оператор даёт OK.

| mode | Семантика |
| --- | --- |
| `EACH` (дефолт) | один issue на каждый непрошедший элемент, в порядке перечисления |
| `ALL` | все элементы обязаны пройти; `summaryIssue: false` → issue на каждый непрошедший; `summaryIssue: true` → ровно один summary-issue, если хоть один не прошёл |
| `COUNT` | пусть *k* = число прошедших; OK ⟺ *k* `op` `value`; при провале — один summary-issue |
| `MIN` | базовый оператор применяется один раз — к минимальному значению элементов |
| `MAX` | то же — к максимальному |

Для `MIN`/`MAX` экстремум берётся по упорядоченному сравнению §2.5; если любой
элемент неклассифицируем или элементы не классифицируются в один вид, экстремум
неопределён, и правило FAIL (один summary-issue): несравниваемое не сравниваем.
`[DR-I]`

**Predicate-режимы.**

| mode | Семантика |
| --- | --- |
| `ANY` (дефолт) | TRUE ⟺ хотя бы один элемент даёт TRUE |
| `ALL` | TRUE ⟺ каждый элемент даёт TRUE |
| `COUNT` | TRUE ⟺ число TRUE-элементов удовлетворяет `op value` |

Результаты `UNDEFINED` считаются не-TRUE. `MIN`/`MAX` с `role: "predicate"` делают
артефакт невалидным.

**`onEmpty`** — поведение, когда wildcard резолвится в ноль элементов:

| onEmpty | check | predicate |
| --- | --- | --- |
| дефолт | `PASS` | `UNDEFINED` |
| `PASS` | правило проходит, без issue | — (невалидно для предиката) |
| `FAIL` | правило фейлится, issue создаётся | — (невалидно) |
| `TRUE` | — (невалидно для check) | `TRUE` |
| `FALSE` | — (невалидно) | `FALSE` |
| `UNDEFINED` | трактуется как `PASS` | `UNDEFINED` |

Опции «аборт на пустом» не существует: жёсткая остановка компонуется из
`onEmpty: "FAIL"` у правила с `level: "EXCEPTION"` — собственные `code` и `message`
аналитика вместо анонимного аборта. `[DR-I]`

Кросс-ролевые значения `onEmpty` (`TRUE`/`FALSE` у check, `PASS`/`FAIL` у
предиката) делают артефакт невалидным; тихой коэрции нет. `[DR-I]`

## 4. Форматы артефактов

### 4.1 Общие правила артефактов

Артефакт — JSON-объект. Каждый артефакт любого типа MUST иметь:

| Поле | Тип | Ограничение |
| --- | --- | --- |
| `id` | строка | непустая; уникальна в пределах снэпшота |
| `type` | строка | одно из `"rule"`, `"condition"`, `"pipeline"`, `"dictionary"` |
| `description` | строка | непустая |

Снэпшот с артефактом, нарушающим эти правила, или с двумя артефактами с одним `id`
— **невалиден** (§4.10). Неизвестные поля верхнего уровня артефакта делают его
невалидным (закрытые схемы: опечатки громкие, неймспейс зарезервирован под будущие
версии спецификации). `[DR-II]`

### 4.2 Идентификаторы, скоупы, видимость

| Форма id артефакта | Видим из |
| --- | --- |
| `library.*` | отовсюду: из любого пайплайна, условия и других library-артефактов |
| `{pipelineId}.*` | только из пайплайна `{pipelineId}` и его условий |
| любой пайплайн, по полному `id` | из `flow` любого другого пайплайна |
| любой словарь, по `id` | из любого правила (словари адресуемы глобально) |

Вложенные пайплайны видимость не наследуют: `a.b.rule_x` виден из пайплайна `a.b`,
не из пайплайна `a`.

*Скоуп* пайплайна — его собственный `id`. *Скоуп* условия выводится из его `id`:
префикс до (не включая) последней точки. `id` условия без точки невалиден.

### 4.3 Резолюция ссылок

Строка-ссылка `r` в шаге или `when`-выражении резолвится так:

| Форма `r` | Резолюция |
| --- | --- |
| начинается с `library.` | абсолютная — как есть |
| содержит `.` | абсолютная — как есть |
| не содержит `.` | скоупированная — раскрывается в `{scope}.{r}` |

Ссылка, не резолвящаяся ни в один артефакт, резолвящаяся в артефакт не того типа
для своей позиции или в артефакт, невидимый из ссылающегося скоупа (§4.2), делает
снэпшот невалидным.

### 4.4 Артефакт `rule`

Поля сверх §4.1:

| Поле | Обязательно | Ограничение |
| --- | --- | --- |
| `role` | да | `"check"` или `"predicate"` |
| `operator` | да | имя встроенного оператора (раздел 3) либо оператора, задекларированного в `requires.operators` (§4.9) |
| `field` | по оператору | dot-notation-путь (§2.7); не используется `any_filled` |
| `fields` | только `any_filled` | непустой массив путей |
| `value`, `value_field`, `flags`, `dictionary` | по оператору | как определено в разделе 3 |
| `aggregate` | опционально | §3.6; осмыслен только при `[*]` в `field` |
| `meta` | опционально | любой JSON-объект; пробрасывается в issues и trace; на исполнение не влияет никогда |

**Check-правила** (`role: "check"`) дополнительно требуют:

| Поле | Ограничение |
| --- | --- |
| `level` | `"WARNING"`, `"ERROR"` или `"EXCEPTION"` |
| `code` | непустая строка; уникальна среди всех check-правил снэпшота |
| `message` | непустая строка |

**Predicate-правила** (`role: "predicate"`) MUST NOT иметь `level`, `code`,
`message`; их присутствие делает артефакт невалидным. Предикат с оператором,
недоступным в предикатной роли (§3.3), невалиден. `aggregate.mode` `MIN` или `MAX`
у предиката невалиден.

### 4.5 Артефакт `condition`

| Поле | Обязательно | Ограничение |
| --- | --- | --- |
| `when` | да | *when-выражение*, см. ниже |
| `steps` | да | непустой массив шагов (§4.8) |

**When-выражение** — одно из:

```
"pred_ref"
{ "all": [ <when-выражение>, … ] }
{ "any": [ <when-выражение>, … ] }
{ "not": <when-выражение> }
```

Массивы `all`/`any` MUST быть непустыми. Листья — ссылки на правила с
`role: "predicate"`; лист, ссылающийся на check-правило, условие или пайплайн,
невалиден. Вложенность не ограничена формой, но ограничена глобальным лимитом
глубины (§2.1).

Семантика — в §5.4.

### 4.6 Артефакт `pipeline`

| Поле | Обязательно | Ограничение |
| --- | --- | --- |
| `entrypoint` | да | boolean, явно |
| `strict` | да | boolean, явно |
| `flow` | да | непустой массив шагов (§4.8) |
| `message` | при `strict: true` | непустая строка; сообщение strict-summary-issue |
| `strictCode` | опционально, только при `strict: true` | непустая строка; дефолт `"STRICT_PIPELINE_FAILED"` |

`entrypoint: true` помечает пайплайн как предполагаемую точку входа и включает
дефолтный выбор (§5.1); он **не** ограничивает вызов — любой пайплайн вызываем по
полному `id`. Семантика `strict` — в §5.7. `message` или `strictCode` у нестрогого
пайплайна невалидны.

**Без поля требований к контексту.** `[D14]` Пайплайны не декларируют
`required_context` (неизвестное поле при закрытых схемах). Полнота runtime-контекста
— явное решение аналитика, выраженное обычными правилами на путях `$context.*` —
например, `not_empty`-проверкой `$context.currentDate` с уровнем, `code` и
`message`, которые выбирает аналитик, — размещёнными там, где аналитик хочет
гарантию (обычно первым шагом `flow` точки входа). Внимание на взаимодействие со
skip-семантикой `[D13]`: value-проверки с отсутствующим операндом `$context.*`
скипаются, поэтому сценарий, зависящий от контекста, MUST гардить его явно, если
отсутствие должно быть ошибкой.

**DAG.** Ориентированный граф с рёбрами-шагами `{ "pipeline": … }` MUST быть
ацикличен. Цикл делает снэпшот невалидным. (Ссылки правил и условий циклов
образовать не могут по построению.)

### 4.7 Артефакт `dictionary`

| Поле | Обязательно | Ограничение |
| --- | --- | --- |
| `entries` | да | непустой массив |

Каждая запись — скаляр (строка, число, boolean; `null` невалиден) или объект хотя
бы с одним из полей `code`, `value`. Семантика матчинга — §3.5.

### 4.8 Шаги

Шаг — объект ровно с одним из ключей `rule`, `condition`, `pipeline`, плюс
опциональный `stepId` (непустая строка; пробрасывается в issues и trace, на
исполнение не влияет). Любой иной ключ или больше одного из трёх — невалидно.

| Ключ | Обязан ссылаться на | Резолюция |
| --- | --- | --- |
| `rule` | артефакт `type: "rule"` с `role: "check"` | §4.3, скоупированные ссылки допустимы |
| `condition` | артефакт `type: "condition"` | §4.3, скоупированные ссылки допустимы |
| `pipeline` | артефакт `type: "pipeline"` | только абсолютный `id`; скоупированные ссылки невалидны |

Rule-шаг, ссылающийся на predicate-правило, — **невалиден**: no-op-шаг в сценарии
валидации почти наверняка ошибка автора, а «валидно, но бессмысленно» — плохой
контракт. `[DR-II]`

### 4.9 Формат снэпшота (`formatVersion: 2`)

Снэпшот — единица дистрибуции и единица конформанса. `[D11]`

```json
{
  "format": "jsonspecs-snapshot",
  "formatVersion": 2,
  "specVersion": "1.0.0",
  "sourceHash": "<64 hex-символа в нижнем регистре>",
  "requires": { "operators": ["valid_inn"] },
  "artifacts": [ … ],
  "meta": {
    "projectId": "…",
    "projectTitle": "…",
    "description": "…",
    "rulesetVersion": "…"
  }
}
```

| Поле | Обязательно | Ограничение |
| --- | --- | --- |
| `format` | да | ровно `"jsonspecs-snapshot"` |
| `formatVersion` | да | ровно `2` (целое) |
| `specVersion` | да | полная SemVer-2.0.0-версия спецификации, на которую нацелен снэпшот |
| `sourceHash` | да | SHA-256, hex в нижнем регистре, от канонизации массива `artifacts` по RFC 8785 (JCS) `[D6]` |
| `requires.operators` | опционально | непустой массив уникальных имён не-встроенных операторов, используемых артефактами `[D10]` |
| `artifacts` | да | массив артефактов; порядок значим только через `sourceHash` |
| `meta` | опционально | свободные метаданные проекта; `projectId` и `rulesetVersion`, если присутствуют, переносятся в `ruleset` результата (раздел 6) |

**Принятие по версии.** Реализация декларирует диапазон поддерживаемых значений
`specVersion`. Она MUST отвергать снэпшот с major-версией, которую не реализует, и
SHOULD отвергать minor-версию новее реализуемой. Prerelease-идентификаторы
сравниваются по прецедентности SemVer.

Снэпшот пиннит контракт поведения (`specVersion`), никогда — версию реализации.
Реализации MAY читать implementation-specific-подсказки из `meta`, но MUST NOT
основывать на них принятие. `[DR-II]`

**Замыкание операторов.** Каждый `operator`, названный правилом, MUST быть либо
встроенным (раздел 3), либо перечисленным в `requires.operators`. Правило с
оператором, не являющимся ни тем, ни другим, невалидно. Реализация, не
предоставляющая какой-либо оператор из `requires.operators`, MUST отвергнуть
снэпшот до любого исполнения — идентификатор `OPERATOR_NOT_FOUND` (раздел 6).
`[D10]`

### 4.10 Множество отвержения

Конформная реализация MUST отвергнуть — отказаться исполнять любой пайплайн —
снэпшот с любым из следующего. *Когда* происходит отвержение (загрузка, подготовка,
первый вызов) — ненаблюдаемо и не ограничивается `[D12]`; *что* оно происходит до
любого исполнения — нормативно.

1. Нарушения конверта снэпшота: неверные `format`/`formatVersion`, некорректный
   или неподдерживаемый `specVersion`, отсутствующий/некорректный `sourceHash`,
   `sourceHash`, не совпадающий с пересчитанным JCS-хэшем `artifacts`.
2. Любой артефакт, нарушающий §4.1 (включая дубликат `id`) или свою типовую схему
   (§4.4–§4.8), включая regex-паттерны вне грамматики или лимитов §3.4, невалидные
   `flags`, невалидные комбинации `aggregate` и кросс-ролевые значения `onEmpty`.
3. Дубликат `code` среди check-правил.
4. Любая неразрешённая, неправильно типизированная или невидимая ссылка (§4.3),
   включая when-листья не на предикаты и pipeline-шаги со скоупированными
   ссылками.
5. Цикл в графе вызовов пайплайнов.
6. Имя оператора, не являющееся ни встроенным, ни задекларированным в
   `requires.operators`; либо задекларированный обязательный оператор, который
   реализация не предоставляет (`OPERATOR_NOT_FOUND`).
7. Любой артефакт, превышающий максимальную глубину JSON (§2.1).

Две конформные реализации MUST совпадать по членству в этом множестве для каждого
входа. *Тексты* и *гранулярность* диагностик отвержения — информативны; вердикт —
нормативен. `[D7]`

## 5. Семантика исполнения

### 5.1 Вход исполнения и валидация входа

Исполнение — чистая функция кортежа:

```
(snapshot, pipelineId?, payload, context?)
```

Валидация входа идёт в следующем нормативном порядке; первый провал определяет
исход (результат `ABORT`, раздел 6), поэтому реализации совпадают в том, *какая*
ошибка репортится, когда применимо несколько:

1. **Выбор пайплайна.** Если `pipelineId` задан, он MUST именовать
   pipeline-артефакт — иначе `ABORT` с `PIPELINE_NOT_FOUND`,
   `details: { "pipelineId": "<заданный>" }`. Если опущен, MUST существовать ровно
   один пайплайн с `entrypoint: true` — иначе `ABORT` с `PIPELINE_ID_REQUIRED`,
   `details: { "entrypointCount": <n> }`. `details` несут только запрошенный id —
   перечисление доступных пайплайнов вызывающему — задача диагностического
   инструментария, а не рантайм-ошибки. `[D7][DR-II]`
2. **Валидация payload.** `payload` MUST быть JSON-объектом; MUST NOT содержать
   зарезервированные ключи (§2.1) на любой глубине
   (`ABORT DANGEROUS_PAYLOAD_KEY`) и MUST NOT превышать лимит глубины
   (`ABORT PAYLOAD_TOO_DEEP`).
3. **Валидация context.** `context`, если присутствует, MUST быть JSON-объектом
   под теми же правилами ключей и глубины (те же коды `ABORT`).

Все провалы валидации входа — `ABORT` (исполнение не могло быть выполнено).
Отдельной фазы required-context не существует `[D14]`: полнота контекста
проверяется обычными правилами, внутри flow, как всё остальное. Флаттенинг payload
(§2.7) — не шаг валидации: любой JSON-объект флаттенится.

### 5.2 Порядок исполнения

`flow` выбранного пайплайна исполняется последовательно. `condition`-шаг с
истинным гардом исполняет свои `steps` последовательно на месте; `pipeline`-шаг
исполняет `flow` пайплайна по ссылке на месте. Результирующий порядок вычислений
правил — depth-first, слева направо по дереву шагов — *документный порядок*.
Issues появляются в результате в документном порядке; внутри одного вычисления
правила поэлементные issues следуют порядку перечисления wildcard (§3.6.1);
strict-summary — по §5.7. `[D5]`

Реализации MUST NOT переупорядочивать, параллелить или дедуплицировать наблюдаемые
эффекты любым способом, меняющим нормативный результат. (Внутренний параллелизм
допустим при неотличимом результате.)

### 5.3 Rule-шаги

Rule-шаг вычисляет своё check-правило над payload по разделу 3 (включая
skip-при-отсутствии `[D13]` и агрегацию). Каждый провал порождает issue с: `level`,
`code`, `message`, `meta` правила; конкретным `field` (для wildcards — конкретный
сматченный путь, не паттерн; для summary — паттерн); `ruleId`; id непосредственно
объемлющего пайплайна как `pipelineId`; `stepId`, если шаг его декларирует;
`expected`/`actual` по разделу 6. Skip не порождает ничего нормативного.

### 5.4 Condition-шаги

`when`-выражение вычисляется над трёхзначным предикатным слоем, схлопываемым в
booleans на листьях:

1. Лист вычисляет своё predicate-правило по разделу 3 (с агрегацией), давая
   `TRUE`, `FALSE` или `UNDEFINED`; `UNDEFINED` схлопывается в `FALSE` на листе.
2. `all` — конъюнкция, `any` — дизъюнкция, `not` — отрицание схлопнутых
   boolean-результатов.

Предикаты не порождают issues и не имеют наблюдаемых побочных эффектов, поэтому
стратегия вычисления (short-circuit или полная, в любом порядке) ненаблюдаема в
нормативном результате и не ограничивается; trace MAY различаться между
реализациями. `[D7][D12]`

Если гард истинен, `steps` условия исполняются на месте (§5.2); иначе шаг — no-op.

### 5.5 Pipeline-шаги

Pipeline-шаг исполняет `flow` пайплайна по ссылке на месте. Issues, порождённые
внутри, несут id *внутреннего* пайплайна как `pipelineId`. `strict` внутреннего
пайплайна применяется к его собственному поддереву исполнения (§5.7); его флаг
`entrypoint` при вызове шагом эффекта не имеет.

### 5.6 Уровни

| `level` | Issue | Эффект на поток | Вклад в `status` |
| --- | --- | --- | --- |
| `WARNING` | создаётся | нет | `OK_WITH_WARNINGS`, если нет сильнее |
| `ERROR` | создаётся | нет — исполнение продолжается | `ERROR` |
| `EXCEPTION` | создаётся | **всё исполнение немедленно останавливается** — включая оставшиеся шаги всех внешних пайплайнов | `EXCEPTION` |

Накопленные issues всегда сохраняются. Матрица `status`/`control` и статус `ABORT`
— в разделе 6.

### 5.7 Строгие пайплайны

Для пайплайна со `strict: true`, когда его поддерево исполнения завершается — хоть
прохождением до конца, хоть остановкой по `EXCEPTION` внутри — и внутри этого
поддерева (его собственные правила, условия и суб-пайплайны включительно) был
порождён хотя бы один issue уровня `ERROR` или `EXCEPTION`, после всех issues
поддерева добавляется один summary-issue:

```json
{
  "kind": "ISSUE",
  "level": "EXCEPTION",
  "code": "<strictCode или STRICT_PIPELINE_FAILED>",
  "message": "<pipeline.message>",
  "field": null,
  "ruleId": "pipeline:<pipelineId>",
  "pipelineId": "<pipelineId>"
}
```

и всё исполнение останавливается (как при любом `EXCEPTION`). Отдельные правила
внутри строгого пайплайна сохраняют декларированные уровни — эскалирует только
групповой исход. Вложенные строгие пайплайны применяют правило каждый к своему
поддереву; внутренний сработавший добавляет summary первым, остановка
распространяется наружу (внешние строгие, тоже сработавшие, добавляют свои summary
в порядке изнутри-наружу). `[DR-II]`

Коды strict-summary генерируются рантаймом и потому вне правила уникальности
`code` check-правил (§4.4); авторам тем не менее SHOULD избегать коллизий значений
`strictCode` с кодами правил — потребители различают issues по `code`.

## 6. Контракт результата

### 6.1 Конверт результата и JSON-представление

Результат исполнения — JSON-объект:

```json
{
  "status": "OK | OK_WITH_WARNINGS | ERROR | EXCEPTION | ABORT",
  "control": "CONTINUE | STOP",
  "issues": [ … ],
  "ruleset": { … },
  "error": { … },
  "trace": [ … ]
}
```

`error` присутствует ровно тогда, когда `status` — `ABORT` (и `issues` тогда пуст).
`trace` опционален и целиком информативен (§6.9).

**Правила представления.** `[D8]` Результат — чистый JSON. «Нет значения»
выражается **опусканием ключа**, никогда — `null`, с единственным сознательным
исключением `"field": null` у issues, не относимых к одному полю (§6.3).
Нормативное равенство результатов — **структурное равенство JSON**: порядок
элементов массивов значим; порядок ключей объектов — нет; числа сравниваются как
binary64-значения (`1` равно `1.0`); детали сериализации (пробелы, стиль
экранирования, порядок ключей на проводе) — не часть конформанса.

### 6.2 Статус и control

| `status` | Смысл | `control` |
| --- | --- | --- |
| `OK` | нет issues | `CONTINUE` |
| `OK_WITH_WARNINGS` | только `WARNING`-issues | `CONTINUE` |
| `ERROR` | хотя бы один `ERROR`-issue, без `EXCEPTION` | `STOP` |
| `EXCEPTION` | хотя бы один `EXCEPTION`-issue (§5.6, §5.7) | `STOP` |
| `ABORT` | исполнение не могло быть выполнено (§6.7) | `STOP` |

`status` полностью определяется сильнейшим присутствующим уровнем issue (или
абортом); `control` полностью определяется `status`. Оба поля существуют потому,
что потребители часто ветвятся по биту go/no-go, не интерпретируя статусы.

### 6.3 Issues

Каждый issue — объект со следующими полями; колонка «Когда присутствует»
нормативна — поле, которое колонка исключает, MUST быть опущено:

| Поле | Тип | Когда присутствует |
| --- | --- | --- |
| `kind` | `"ISSUE"` | всегда |
| `level` | `WARNING \| ERROR \| EXCEPTION` | всегда |
| `code` | строка | всегда — авторское (`rule.code`, `strictCode`) |
| `message` | строка | всегда — авторское (`rule.message`, `pipeline.message`); нормативно как проброс данных `[D7]` |
| `field` | строка или `null` | всегда; конкретный разрезолвленный путь для полевых issues (для wildcard-элементов — конкретный сматченный путь вроде `x[2].v`); паттерн (`x[*].v`) для групповых summary; `null` для issues, не относимых к одному полю (`any_filled`, strict-summary) |
| `ruleId` | строка | всегда; `id` правила либо `pipeline:<pipelineId>` для strict-summary |
| `pipelineId` | строка | всегда; непосредственно объемлющий пайплайн (§5.5) |
| `stepId` | строка | только когда породивший шаг его декларирует |
| `expected` | значение | по §6.4 |
| `actual` | значение | по §6.4; опущено, когда единого actual-значения нет |
| `details` | объект | только групповые summary-issues (§6.5); нормативные машиночитаемые факты issue, зеркально `error.details` при `ABORT` (§6.7) — один паттерн на весь контракт: `code` идентифицирует класс, `details` несёт факты класса |
| `meta` | объект | когда правило декларирует `meta` (§6.6) |

Порядок issues в `issues[]` нормативен: документный порядок вычислений правил
(§5.2), порядок перечисления wildcard внутри правила (§3.6.1), strict-summary по
§5.7.

### 6.4 `expected` и `actual`

| Класс операторов | `expected` | `actual` |
| --- | --- | --- |
| сравнения со значением (`equals`, `not_equals`, `contains`, `matches_regex`, `not_matches_regex`, `greater_than`, `less_than`, `length_*`) | `value` правила дословно | разрезолвленное значение поля |
| проверки типа (`is_boolean`, `is_string`, `is_number`, `is_integer`) | опущен | разрезолвленное значение поля |
| `not_empty` | опущен | опущен при отсутствии; значение (`null` или `""`) при присутствующем-но-пустом — как JSON-значение, т.е. `"actual": null` здесь легален как *значение* actual, в отличие от опускания ключа |
| `is_empty`, `not_true` | опущен | разрезолвленное значение поля |
| словарные операторы | объект `dictionary` правила дословно (`{"type":"static","id":…}`) | разрезолвленное значение поля |
| `field_*_field` | разрезолвленное **значение** `value_field` `[DR-III]` | разрезолвленное значение `field` |
| `any_filled` | опущен | опущен |

Поскольку флаттенинг допускает в листьях только скаляры и пустые контейнеры
(§2.7), значения `expected`/`actual` всегда мелкие; их обрезание или транспортная
нормализация не определяются и не допускаются.

### 6.5 Групповые summary-issues

`[DR-III]` Summary-issue (порождаемый `ALL` c `summaryIssue: true`, проваленным
`COUNT`, `MIN`/`MAX` либо `onEmpty: "FAIL"`) несёт `field` = wildcard-паттерн и
объект `details`:

| Источник | `details` | `expected` / `actual` |
| --- | --- | --- |
| `ALL`, `summaryIssue: true` | `{"mode":"ALL","total":<n>,"failed":<k>}` | опущены / опущены |
| Провал `COUNT` | `{"mode":"COUNT","op":"…","value":<v>,"total":<n>,"passed":<k>}` | опущены / опущены |
| Провал `MIN` / `MAX` | `{"mode":"MIN"\|"MAX"}` | по §6.4 для базового оператора; `field` — **конкретный путь элемента-экстремума**, не паттерн; `actual` = значение экстремума. Если экстремум неопределён (§3.6.2), `field` — паттерн, `expected`/`actual` опущены |
| `onEmpty: "FAIL"` | `{"mode":"<эффективный режим>","total":0}` | опущены / опущены |

Правило потребителя — одно предложение: **есть `details.mode` ⇒ групповой вердикт,
читай `details`; иначе — обычная value-issue, читай `expected`/`actual`.**
Поэлементные issues (`EACH` или `ALL` с `summaryIssue: false`) — обычные issues
§6.3/§6.4 и никогда не несут `details`.

### 6.6 Проброс `meta`

Когда правило декларирует `meta`, каждый issue этого правила несёт `meta`
**дословно как у автора**. Рантайм MUST NOT добавлять, удалять или переписывать
ключи внутри; рантаймовые факты живут в `details` (§6.5) или нигде. `[DR-III]`

### 6.7 `ABORT` и два канала отказа

Каналов отказа два; их смешение — ошибка конформанса:

**Канал A — отвержение снэпшота (§4.10).** Снэпшот отвергается до любого
исполнения. *Вердикт* нормативен; форма репортинга (список диагностик, брошенная
ошибка, exit-код) — на усмотрение реализации. Одна причина отвержения имеет
нормативный идентификатор, потому что зависит от окружения: `OPERATOR_NOT_FOUND` —
`requires.operators` снэпшота именует оператор, который реализация не
предоставляет. `[D10]`

**Канал B — `ABORT` исполнения.** Кортеж был принят к исполнению, но исполнение не
могло быть выполнено. Результат несёт:

```json
"error": { "code": "…", "message": "…", "details": { … } }
```

`code` и `details` нормативны; `message` — информативный свободный текст. `[D7]`
Нормативный enum кодов и точная форма `details` по кодам:

| `code` | Когда | `details` |
| --- | --- | --- |
| `PIPELINE_NOT_FOUND` | §5.1 шаг 1 | `{"pipelineId": "<запрошенный>"}` |
| `PIPELINE_ID_REQUIRED` | §5.1 шаг 1 | `{"entrypointCount": <n>}` |
| `DANGEROUS_PAYLOAD_KEY` | §5.1 шаги 2–3 | `{"path": "<dot-путь>", "key": "<ключ>"}`; при нескольких зарезервированных ключах репортится **лексикографически наименьший** путь (по кодовым точкам) — это делает `details` детерминированным при любом порядке обхода `[DR-III]` |
| `PAYLOAD_TOO_DEEP` | §5.1 шаги 2–3 | `{"maxDepth": 256}` — и ничего больше: *какой* путь первым превысит лимит, зависит от порядка обхода, поэтому путь не появляется ни в одной нормативной поверхности (информативный `message` MAY его нести) `[DR-III]` |
| `OPERATOR_FAULT` | реализация оператора бросила исключение/упала при исполнении | `{"ruleId": "…", "operator": "…"}` |
| `OPERATOR_CONTRACT_VIOLATION` | оператор вернул значение вне своей декларированной формы результата (§3.1) | `{"ruleId": "…", "operator": "…"}` |

Enum закрыт для этой версии спецификации: реализации MUST NOT эмитить иные коды в
канале B. Встроенные операторы, реализованные корректно, никогда не порождают два
кода `OPERATOR_*`; на входах, валидных по этой спецификации, они невозможны вовсе
— фикстуры покрывают их только через намеренно сломанные кастомные операторы.

### 6.8 Провенанс `ruleset`

```json
"ruleset": {
  "specVersion": "1.0.0",
  "sourceHash": "…",
  "projectId": "…",
  "rulesetVersion": "…",
  "engineVersion": "…"
}
```

| Поле | Присутствие | Поверхность |
| --- | --- | --- |
| `specVersion` | всегда — эхо снэпшота | нормативно `[DR-III]` — потребитель логирует, по какому контракту поведения получен результат, не держа снэпшот |
| `sourceHash` | всегда — эхо снэпшота | нормативно |
| `projectId`, `rulesetVersion` | когда присутствуют в `snapshot.meta` | нормативный проброс |
| `engineVersion` | опционально | **информативно** — идентификатор реализации, исключён из сравнения конформанса `[D7]` |

### 6.9 Trace

`trace` целиком информативен. `[D7]` Реализация MAY его опускать, порождать в
любой гранулярности или делать опциональным per-call. Когда порождается, ему
SHOULD быть массивом объектов-событий как минимум с `type` и идентификаторами
артефактов, и SHOULD фиксировать value-семантичные скипы (`[D13]`) и исходы гардов
условий — но никакое сравнение конформанса его не читает, а таймстампы (`at`)
делают побайтовое сравнение бессмысленным by design. Trace MUST NOT быть
единственным носителем какого-либо нормативного факта.

## 7. Конформанс

### 7.1 Декларация конформанса

Декларация конформанса называет: реализацию и версию; поддерживаемый диапазон
`specVersion` (§4.9); множество имён не-встроенных операторов, предоставляемых
инсталляцией. Конформанс двух реализаций оценивается **относительно равных
множеств зарегистрированных операторов**: множество отвержения (§4.10)
параметризовано этим множеством через `requires.operators` — один и тот же снэпшот
легитимно принят инсталляцией, предоставляющей `valid_inn`, и отвергнут
(`OPERATOR_NOT_FOUND`) той, что не предоставляет. Для снэпшотов только на
встроенных операторах конформанс безусловен. `[D10]`

### 7.2 Нормативная проекция

Конформанс сравнивает *нормативную проекцию* поведения:

1. вердикт по снэпшоту: принят или отвергнут (§4.10), плюс идентификатор
   `OPERATOR_NOT_FOUND`, где применим;
2. для принятых снэпшотов и каждого кортежа исполнения: `status`, `control`,
   `issues[]` целиком (каждое поле §6.3, в порядке), `error.code` +
   `error.details` и `ruleset` без `engineVersion`.

Исключено из сравнения: `trace`, `engineVersion`, `error.message`, тексты и
гранулярность диагностик отвержения и всё, что реализация эмитит вне объекта
результата. Сравнение — структурное (§6.1).

### 7.3 Conformance-фикстуры

Дерево `fixtures/` репозитория `jsonspecs/spec` — **нормативное приложение**:
текст и фикстуры версионируются вместе, атомарно, одним тегом. Два вида фикстур:

```json
// evaluation-фикстура
{
  "name": "d02/…",
  "snapshot": { … },
  "operators": ["…"],            // зарегистрированное множество для фикстуры, дефолт []
  "input": { "pipelineId": "…", "payload": { … }, "context": { … } },
  "expected": {
    "status": "…", "control": "…",
    "issues": [ … ],
    "ruleset": { "specVersion": "…", "sourceHash": "…" }
  }
}

// rejection-фикстура
{
  "name": "d04/…", "snapshot": { … }, "operators": [],
  "expected": { "verdict": "reject", "identifier": "OPERATOR_NOT_FOUND"? }
}
```

Прогонщик исполняет каждую фикстуру против реализации и структурно сравнивает
нормативную проекцию. Реализация конформна версии спецификации X **тогда и только
тогда, когда проходит каждую фикстуру версии X**. Фикстуры организованы по
решениям и семантическим областям (`d01-numbers/`, `d04-regex/`, `semantics/`, …);
каждое решение D1–D14 покрыто как минимум одной фикстурой.

### 7.4 Сводка требований

Конформная реализация:

- MUST принимать и отвергать ровно множество §4.10, относительно своих
  зарегистрированных операторов;
- MUST выдавать идентичную нормативную проекцию для каждого кортежа исполнения;
- MUST проходить полный набор фикстур декларируемой версии спецификации;
- MUST NOT требовать какого-либо входа, флага или режима сверх кортежа исполнения
  для достижения перечисленного (конформанс — поведение по умолчанию, а не опция);
- MAY делать что угодно, не наблюдаемое через нормативную проекцию: компилировать
  или интерпретировать, кэшировать, параллелить, порождать любой trace,
  экспонировать любой API. `[D12]`

### 7.5 Вне скоупа

Граница в одном месте: API и сигнатуры функций; момент валидации; стратегия
компиляции и диагностика сверх вердикта; производительность и ресурсные лимиты
сверх нормативных (§2.1, §3.4.2); транспортное обрезание результатов; поведение
кастомных операторов `[D10]`; содержимое trace; legacy-поверхности прототипа
(`engine.minVersion`, `paths[]`, `payload.__context`, `required_context`)
`[D11][D14]`.
