# Переход с 1.0.0-rc.5 на 1.0.0-rc.6

English version: [MIGRATION_RC6.md](MIGRATION_RC6.md).

RC.6 меняет правила обхода `[*]`. Поля языка описания правил не добавляются и не
удаляются. Изменение намеренно несовместимо для правил с `[*]`: одно и то же правило
может дать другой результат, если у элемента массива нет точного продолжения пути после
последнего `[*]`.

## Обязательные изменения пакета правил

1. Установите в зафиксированной сборке `specVersion: "1.0.0-rc.6"`.
2. Пересоберите её целиком и заново вычислите `ruleset.sourceHash`. Сборка для RC.5 не
   становится сборкой RC.6, даже если исходные файлы пакета не менялись.
3. Выполните весь набор проверок совместимости RC.6. Особенно важны все примеры из
   `fixtures/d31-wildcard/`.
4. Повторно выполните примеры пакета, в правилах которых есть `[*]`. Проверьте изменения
   найденных ошибок, счётчиков, конкретных путей и итогов агрегации.

Может ли одна версия движка принимать сборки RC.5 и RC.6 одновременно, решает разработчик
движка. Спецификация поведения RC.6 не требует совместимости с RC.5 и не задаёт запасного
поведения между версиями спецификации.

## Что изменилось

В RC.5 совпадения для `[*]` выводились из существующих листьев плоского представления
путей. Поэтому путь `order.items[*].sku` мог пропустить элемент массива без поля `sku`.

В RC.6 каждый существующий индекс массива, достигнутый через `[*]`, создаёт структурного
кандидата. Если точное продолжение пути после последнего `[*]` отсутствует или по нему
нельзя пройти, сохраняется один отсутствующий кандидат с конкретным путём:

```text
order.items[0].sku = отсутствует
order.items[1].sku = "BKS-0987"
```

Если точный сегмент отсутствует или непроходим до последнего `[*]`, прекращается только
эта ветвь: в ней нет массива, из которого следующий `[*]` мог бы получить кандидатов.
Подробности приведены в `SPEC_RU.md` §3.6.1 и решении D31.

Тип точного сегмента пути имеет значение:

- сегмент-ключ проходит только по собственному полю объекта JSON;
- сегмент-индекс проходит только к существующему элементу массива JSON;
- поле объекта с именем `"0"` не является индексом массива `[0]`;
- индекс за границами массива означает отсутствие значения.

## Обязательное поле каждого элемента

Следующее правило теперь проверяет `sku` у каждого существующего элемента
`order.items`:

```json
{
  "type": "rule",
  "operator": "not_empty",
  "field": "order.items[*].sku",
  "aggregate": {
    "mode": "ALL",
    "issueMode": "EACH",
    "onEmpty": "FAIL"
  },
  "issue": {
    "level": "ERROR",
    "code": "ORDER.ITEM.SKU.REQUIRED",
    "message": "Укажите артикул товара"
  }
}
```

Для элемента без `sku` оператор `not_empty` возвращает `FAIL`. Режим `EACH` создаёт
найденную ошибку с достроенным конкретным путём, например `order.items[0].sku`.

Пустая или отсутствующая коллекция, значение другого типа и иной непроходимый путь не
создают структурных кандидатов. Этот случай обрабатывает `onEmpty`. Он отличается от
существующего элемента массива, у которого нет обязательного дочернего поля.

## Операторы значений и счётчики

К отсутствующему структурному кандидату применяется обычная семантика отсутствия
конкретного оператора. Например, `equals` возвращает для него `SKIP`. Поэтому:

- кандидаты существуют, и `onEmpty` не применяется;
- если все кандидаты дали `SKIP`, итог агрегации равен `SKIP`;
- `matched` считает всех структурных кандидатов;
- `evaluated` считает только `PASS` и `FAIL`;
- `skipped` считает результаты `SKIP`.

Если само отсутствие должно дать ошибку, используйте оператор проверки наличия, например
`not_empty`. Оператор сравнения со значением и `onEmpty` не превращают отсутствующее
дочернее поле в `FAIL`.

## Путь, оканчивающийся на `[*]`

Для пути, который оканчивается на `[*]`, например `order.items[*]`, после формирования
структурных кандидатов по-прежнему применяется определение листа из `SPEC_RU.md` §2.7.
Непустой объект или массив сам по себе не является операндом-листом. Такие правила нужно
проверить отдельно: RC.6 не превращает их в проверку элемента массива целиком.

## Оператор только для проверки совместимости

Зарезервированный оператор `conformance.rule.tri` и его вход `"INVALID"` предназначены
только для проверочных примеров. При `"INVALID"` оператор возвращает недопустимое значение.
Так движок доказывает, что поздние кандидаты `[*]` действительно вычисляются. Это не
возможность языка бизнес-правил, и её нельзя предоставлять аналитикам.
