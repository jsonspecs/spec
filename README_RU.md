# jsonspecs/spec

Спецификация поведения и conformance-набор для рантаймов правил JSONSpecs.
English version: [README.md](README.md).

Репозиторий определяет, **что** делает рантайм правил — и никогда **как**. Любая
реализация (Node.js, Java, Go, …), которая на одном и том же снэпшоте, payload и
context выдаёт тот же нормативный результат и принимает/отвергает то же множество
входов, — конформна. API, стратегия компиляции, внутренние представления и
производительность — явно вне скоупа.

## Состав

| Путь | Роль |
| --- | --- |
| [`SPEC.md`](SPEC.md) / [`SPEC_RU.md`](SPEC_RU.md) | **Канон.** Спецификация поведения версии, соответствующей тегу репозитория. Описывает только ожидаемое поведение; аргументация — в реестре решений. |
| [`DECISIONS.md`](DECISIONS.md) / [`DECISIONS_RU.md`](DECISIONS_RU.md) | Реестр решений D1–D14 + аддендумы: каждое проектное решение с обоснованием, альтернативами и миграционной ценой; спека ссылается на них как `[D#]` / `[DR-*]`. |
| [`fixtures/`](fixtures/) | **Нормативное приложение.** Conformance-фикстуры; реализация конформна тогда и только тогда, когда проходит все. См. `fixtures/README.md`. |
| [`source/`](source/) | Историческое, ненормативное: спека прототипа (`jsonspecs/rules`) и аудит боевого пакета, на которые ссылаются решения. |
| [`tools/`](tools/) | Генератор и валидатор фикстур (Node ≥ 20, без зависимостей). |

## Как потреблять спеку разработчику реализации

1. Реализуйте поведение из `SPEC.md`. Там, где утверждение помечено `[D#]`/`[DR-*]`,
   реестр объясняет *почему* — прочитайте его прежде, чем решить, что спека ошибается.
2. Соберите прогонщик фикстур: для каждого файла в `fixtures/**` подайте
   `snapshot` + `input` в свой рантайм и сравните **нормативную проекцию**
   (SPEC.md §7.2) с `expected` структурным равенством JSON (§6.1). Rejection-фикстуры
   обязаны отвергаться до любого исполнения.
3. Задекларируйте конформанс: реализация + версия, поддерживаемый диапазон
   `specVersion`, имена зарегистрированных не-встроенных операторов (§7.1).

**Не** считайте эталоном никакую существующую реализацию. Эталон — фикстуры; где
реализация расходится с фикстурами, неправа реализация (или фикстура — заводите
issue, набор версионируем и правится до тега).

## Версионирование и релизный процесс

- Спецификация следует **SemVer 2.0.0**. Текст и фикстуры версионируются вместе,
  атомарно, одним тегом: `vMAJOR.MINOR.PATCH[-rc.N]`.
- **Patch** — редакторские правки, фикстуры на уже специфицированное поведение.
  **Minor** — обратно совместимые дополнения поведения (новые операторы, новые поля
  с определённой семантикой отсутствия). **Major** — всё, что меняет вердикт на
  существующем валидном входе.
- Текущий статус: **1.0.0-rc.2**. Тег `v1.0.0` вешается после стендового сравнения
  реализаций (Node v3 vs Java), подтверждающего набор:

```
git tag v1.0.0-rc.2 && git push origin v1.0.0-rc.2   # релиз-кандидат (prerelease)
git tag v1.0.0      && git push origin v1.0.0        # после подтверждения стендом
```

Пуш тега `v*` запускает `.github/workflows/release.yml`: валидация фикстур, сборка
архива исходников, создание GitHub Release с автогенерируемыми заметками (теги с
`-rc` помечаются prerelease). CI (`ci.yml`) дополнительно гарантирует, что
`fixtures/` и `tools/generate-fixtures.mjs` не расходятся.

## Локальные проверки

```
node tools/validate-fixtures.mjs    # структурная валидность + целостность sourceHash
node tools/generate-fixtures.mjs    # перегенерация фикстур (правьте генератор, не JSON)
```

## Связь с jsonspecs/rules

`jsonspecs/rules` (Node) — прикладной прототип, из опыта которого выросла эта
спецификация; его документы сохранены дословно в `source/`. Начиная с
`@jsonspecs/rules` v3, Node-реализация декларирует конформанс этой спеке наравне с
любой другой — особого статуса у неё нет.
