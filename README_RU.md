# Спецификация поведения и conformance-набор для рантаймов правил JSONSpecs

English version: [README.md](README.md).

Репозиторий определяет, **что** делает рантайм правил — и никогда **как**. Для снэпшотов только со
встроенными операторами конформна любая реализация (Node.js, Java, Go, …), которая на одних и тех же
входах выдаёт тот же нормативный результат и так же принимает или отвергает снэпшоты. Равенство сквозного
поведения пользовательских операторов дополнительно требует межрантаймового профиля пакета операторов из `SPEC.md` §7.1.
API, стратегия компиляции, внутренние представления и производительность явно вне области спецификации.

## Состав

| Путь                                                                  | Роль                                                                                                                                                           |
| --------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| [`SPEC.md`](SPEC.md)                                                  | **Канон.** Нормативная спецификация поведения версии, соответствующей тегу репозитория.                                                                        |
| [`SPEC_RU.md`](SPEC_RU.md)                                            | Полный русский перевод с той же структурой; при расхождении нормативен `SPEC.md`.                                                                              |
| [`DECISIONS.md`](DECISIONS.md) / [`DECISIONS_RU.md`](DECISIONS_RU.md) | Реестр решений D1–D31 и дополнений: каждое проектное решение с обоснованием, альтернативами и ценой миграции; спецификация ссылается на них как `[D#]` / `[DR-*]`. |
| [`MIGRATION_RC6.md`](MIGRATION_RC6.md) / [`MIGRATION_RC6_RU.md`](MIGRATION_RC6_RU.md) | Обязательные изменения пакета правил и различия поведения при переходе с RC.5 на RC.6. |
| [`fixtures/`](fixtures/)                                              | **Нормативное приложение.** Прохождение всех conformance-фикстур необходимо, но недостаточно для конформанса. См. `fixtures/README.md`.                        |
| [`source/`](source/)                                                  | Историческое, ненормативное: спека прототипа (`jsonspecs/rules`) и аудит боевого пакета, на которые ссылаются решения.                                         |
| [`tools/`](tools/)                                                    | Генератор и валидатор фикстур (Node ≥ 20, без зависимостей).                                                                                                   |

## Как потреблять спеку разработчику реализации

1. Реализуйте поведение из `SPEC.md`. Там, где утверждение помечено `[D#]`/`[DR-*]`,
   реестр объясняет _почему_ — прочитайте его прежде, чем решить, что спека ошибается.
2. Соберите прогонщик фикстур: для каждого файла в `fixtures/**` подайте
   `snapshot` + `input` в свой рантайм и сравните **нормативную проекцию**
   (SPEC.md §7.2) с `expected` структурным равенством JSON (§6.1). Rejection-фикстуры
   обязаны отвергаться до любого исполнения. Отсутствие `identifier` в ожидаемом
   отвержении нужно проверять, а не игнорировать. Сравнивайте JSON-данные, а не
   прототипы, классы, внутреннее представление объектов или порядок обхода полей в языке реализации.
3. Задекларируйте конформанс: реализация + версия, поддерживаемый диапазон
   `specVersion`, имена зарегистрированных не-встроенных операторов (§7.1).

**Не** считайте эталоном никакую существующую реализацию. Канон — английский
`SPEC.md`; фикстуры служат его исполняемым нормативным приложением. Противоречие
исправляется erratum к спеке и версионированной правкой фикстуры, а не молчаливым
следованием реализации.

## Версионирование и релизный процесс

- Спецификация следует **SemVer 2.0.0**. Текст и фикстуры версионируются вместе,
  атомарно, одним тегом: `vMAJOR.MINOR.PATCH[-rc.N]`.
- **Patch** — редакторские правки, фикстуры на уже специфицированное поведение.
  **Minor** — обратно совместимые дополнения поведения (новые операторы, новые поля
  с определённой семантикой отсутствия). **Major** — всё, что меняет вердикт на
  существующем валидном входе.
- Текущий статус: **1.0.0-rc.6**. Тег `v1.0.0` ставится после стендового сравнения
  реализаций (Node v3 vs Java), подтверждающего набор:

```
git tag v1.0.0-rc.6 && git push origin v1.0.0-rc.6   # релиз-кандидат
git tag v1.0.0      && git push origin v1.0.0        # после подтверждения стендом
```

Пуш тега `v*` запускает `.github/workflows/release.yml`: валидация фикстур, сборка
архива исходников, создание GitHub Release с автогенерируемыми заметками (теги с
`-rc` помечаются prerelease). CI (`ci.yml`) дополнительно гарантирует, что
`fixtures/` и `tools/generate-fixtures.mjs` не расходятся.

## Локальные проверки

```
node tools/validate-fixtures.mjs    # структурная валидность + целостность sourceHash
node tools/validate-doc-parity.mjs  # паритет структуры английской и русской SPEC
node tools/generate-fixtures.mjs    # перегенерация фикстур (правьте генератор, не JSON)
java tools/JcsUtf16Check.java       # независимый Java-вектор hash/order
```

## Связь с jsonspecs/rules

`jsonspecs/rules` (Node) — прикладной прототип, из опыта которого выросла эта
спецификация; его документы сохранены дословно в `source/`. Начиная с
`@jsonspecs/rules` v3, Node-реализация декларирует конформанс этой спеке наравне с
любой другой — особого статуса у неё нет. Rules v3 соответствует RC.5; Rules v4 —
первая версия Node-реализации, предназначенная для RC.6.
