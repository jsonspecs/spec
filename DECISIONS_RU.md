# JSONSpecs Contract Spec — реестр решений

Перевод нормативного английского реестра [DECISIONS.md](DECISIONS.md). При
расхождениях нормативен английский текст.

Цель контракта: любые две конформные реализации на одном снэпшоте и одном payload обязаны выдать
одинаковый нормативный результат. Каждое решение ниже — точка, где без явной фиксации
реализации разойдутся. Формат: контекст → варианты → рекомендация → цена против текущего
Node-поведения.

**Текущее состояние: все 30 решений (D1–D30) и аддендумы [DR-I]–[DR-VIII] —
`APPROVED`. Реестр — вход для текста спеки и conformance-фикстур; спека ссылается
сюда как `[D#]` / `[DR-*]`.** По D2 владельцем подтверждено: фикс
Node-реализации (code points вместо UTF-16 units) — приемлемая цена мультиплатформенности.
Критерий закрытия D4, D10, D11 — минимальный суммарный компромисс для мейнстримных
бэкенд-платформ (JVM, Go, .NET, Python, Rust, PHP, Node), а не удобство первой
Node-реализации.

## D1. Числовая модель — APPROVED; оговорка о точности заменена D23

**Контекст.** `equals`, `greater_than`, `field_*_field`, `is_integer`, `COUNT`-агрегация — всё
опирается на понятие «число». Сейчас неявно: IEEE 754 binary64 (Node). Java может парсить в
`BigDecimal`, Go — в `float64` или `json.Number`, и `equals` на границах точности разойдётся.

**Варианты.**

| Вариант | Плюсы | Минусы |
|---|---|---|
| (a) IEEE 754 binary64 | Совпадает с Node; есть везде; JSON де-факто | Потеря точности за пределами 2^53; денежные суммы в копейках безопасны, в дробях — нет |
| (b) Decimal (arbitrary precision) | Точность для денег | Расходится с Node (breaking); в Go/JS нет нативно; парсер чисел усложняется |
| (c) Гибрид int64 + binary64 | — | Худшее из обоих: правила сравнения int↔float надо определять отдельно |

**Рекомендация: (a) binary64**, с двумя нормативными оговорками:

1. Целые в диапазоне ±(2^53 − 1) представляются точно; артефакты и payload с числами вне этого
   диапазона или с дробями, непредставимыми в binary64, — вне гарантий контракта
   (реализация MAY давать любой из корректных результатов округления).
2. Числовые строки (для date/number-сравнений) парсятся по уже зафиксированной регулярке
   `^[+-]?\d+(\.\d+)?([eE][+-]?\d+)?$` в binary64.

Следствие для `equals`: `1 === 1.0` (одно значение binary64), `1 !== "1"` (тип не совпал) —
совпадает с текущим поведением.

**Цена:** нулевая, фиксируем статус-кво словами, не зависящими от JS.

## D2. Единица длины строки (`length_equals`, `length_max`) — APPROVED

**Контекст.** Сейчас `String(field).length` = UTF-16 code units. Go нативно считает байты или
руны. `"Пётр"` везде даст 4, а `"👍"` даст 2 (Node/Java UTF-16), 1 (code points), 4 (UTF-8 bytes).

**Варианты.**

| Вариант | Оценка |
|---|---|
| (a) UTF-16 code units | Нативно для Node/Java; для Go — искусственная конвертация; семантически «почему 16?» |
| (b) Unicode code points | Кросс-языково наименее сюрпризно; для кириллицы (банковские ФИО/адреса) = интуиции аналитика; в Go — нативные руны |
| (c) UTF-8 bytes | Кириллица ×2 — прямо опасно для банковских правил длины |
| (d) Grapheme clusters | Зависит от версии Unicode и библиотеки → недетерминизм by design |

**Рекомендация: (b) code points.** (c) и (d) отпадают сразу. Между (a) и (b): расхождение (a)/(b)
существует только на суррогатных парах (emoji, редкие CJK), которых в банковских payload
практически нет, поэтому миграционный риск минимален, а определение чище.

**Цена:** формально breaking против Node на строках с суррогатными парами. Требуется фикс
Node-реализации (`[...str].length` вместо `str.length`) + фикстуры именно на этот случай.

## D3. `length_*` над нестроками — APPROVED

**Контекст.** Сейчас `String(field)` — JS-стрингификация: `1e21` → `"1e+21"`, `true` → `"true"`.
Воспроизводить это в Java/Go — переносить ECMAScript `Number::toString` в контракт.

**Варианты:** (a) FAIL для любого нестрокового значения; (b) нормативная каноническая
стрингификация.

**Рекомендация: (a) FAIL.** Проверка длины числа — почти всегда ошибка автора правила
(хочешь длину — храни строкой; хочешь диапазон — `greater_than`/`less_than`). Вариант (b) тащит
в контракт самую капризную часть JS.

**Цена:** breaking, если в существующих rule-паках есть `length_*` над числовыми полями.
Нужен аудит паков перед фиксацией (grep по операторам + типам полей в тестовых payload).

## D4. Regex: диалект — APPROVED; флаги и препроцессинг удалены D29

Ниже сохранено исходное решение rc.1. В актуальной версии 1 действуют диалект и два
regex-оператора, но флагов и дополнительного схлопывания обратных косых черт нет.

**Поправка (утверждена по итогам аудита боевого пакета):** во встроенный пакет операторов
добавляется check-оператор `not_matches_regex` — тот же контракт паттерна и препроцессинга,
что у `matches_regex`, инвертированный вердикт. Поведение при отсутствии поля: **FAIL**
(поле должно присутствовать и не соответствовать паттерну). Кросс-языковая цена нулевая —
инверсия результата того же RE2-матча. Мотивация: все lookahead-паттерны боевого пакета
(17 правил) — форма «формат X и не исключение Y», т.е. отсутствие отрицания в операторах,
а не потребность в мощности regex. Той же логикой во встроенные переносится предикат
`not_in_dictionary` (отрицание встроенного `in_dictionary`; поведение при отсутствии поля —
симметрично `in_dictionary`, зафиксировать в спеке явно).

**Контекст.** `matches_regex` = `new RegExp()` — ECMAScript-диалект. Go stdlib — RE2 (нет
backreferences и lookaround). Java — третий диалект. Плюс идиосинкразия: одноразовый проход
схлопывания `\\` → `\` перед компиляцией паттерна.

**D4.1 Диалект — RE2-совместимое подмножество.**

Рыночная матрица (может ли штатный движок языка исполнить диалект без сторонних зависимостей):

| Язык / платформа | Штатный движок | Исполнит RE2-подмножество | Исполнит полный ECMAScript |
|---|---|---|---|
| Node.js / TS | V8 (ECMAScript) | да (надмножество) | да |
| Java / Kotlin | java.util.regex | да (надмножество) | почти, но нюансы семантики |
| Go | regexp (RE2) | **да, нативно** | **нет** — только сторонний движок |
| C# / .NET | Regex (+ NonBacktracking с .NET 7) | да (NonBacktracking — почти изоморфен RE2) | почти |
| Python | re | да (надмножество) | почти |
| Rust | crate `regex` (стандарт де-факто) | **да, нативно** | **нет** — только `fancy-regex` и аналоги |
| PHP | PCRE2 | да (надмножество) | почти |

Вывод однозначный: RE2-подмножество — единственный вариант, при котором **ни один**
мейнстримный бэкенд-язык не обязан тащить нестандартный движок. Половина языков исполняет
его нативно, вторая половина — как подмножество своего движка. Обратное неверно: полный
ECMAScript выбивает Go и Rust из штатных средств. Бонус: линейное время исполнения — ReDoS
закрывается конструктивно, эвристический линтер из текущей реализации становится не нужен.

Нормативный whitelist: литералы, `[]`-классы и отрицания, `|`, якоря `^ $`, квантификаторы
`* + ? {n} {n,} {n,m}`, группы `(...)` и `(?:...)`, escape-классы `\d \w \s \D \W \S`
с **ASCII-семантикой** (`\d` = `[0-9]`; это дефолт JS и Go; Rust по умолчанию трактует `\d`
как Unicode — реализация обязана компенсировать флагом `(?-u)` или транспиляцией в явные
классы, это строчка кода). Запрещены: backreferences, lookaround `(?= (?! (?<= (?<!`,
именованные группы, `\b` (семантика word boundary расходится между движками: ASCII в RE2/JS,
Unicode в Rust; для банковских форматов не нужен — паттерны якорятся `^...$`; добавление
в подмножество позже — обратно совместимо, удаление — нет).

Механика конформанса: спека поставляет нормативную грамматику подмножества (EBNF) + фикстуры
«валидный/невалидный паттерн». Валидатор паттернов — ~200 строк на любом языке, паттерн вне
подмножества → снэпшот отвергается (входит в нормативное множество reject'ов из D7).

**D4.2 Флаг `i` — Unicode simple case folding, culture-invariant.** Совпадает с нативным
поведением RE2 (Go/Rust) и ECMAScript в `u`-режиме (Node обязан компилировать с флагом `u`).
Java — `CASE_INSENSITIVE | UNICODE_CASE`, .NET — обязательно `CultureInvariant` (иначе
турецкая локаль ломает `i`/`I`). Для кириллицы (`и`/`И`) simple folding везде идентичен.
ASCII-only folding отклонён: молча ломает `i` на кириллических паттернах — худший вид
сюрприза для банковского контекста.

**D4.3 Препроцессинг `\\` → `\` — зафиксировать нормативно (вариант b).** Ключевой аргумент
через призму «цена для рынка»: препроцессинг — одна строка `replace` на любом языке,
кросс-языковая цена нулевая. Цена варианта (a) — миграция всех существующих артефактов
с `^\\d{6}$`-стилем и риск тихой смены семантики паттернов при миграции. Фиксируем (b)
в контракте v1; если когда-нибудь появится formatVersion 2 артефактов — там можно перейти
на чистый JSON-escaping без прохода, миграцию сделает инструмент.

## D5. Порядок обхода wildcard и тотальный порядок `issues[]` — APPROVED; старая формулировка заменена D26/D29

**Контекст.** «Один и тот же результат» = в том числе одинаковый порядок в `issues[]`.
Порядок шагов детерминирован `flow`, но `accounts[*].balance` перечисляет совпадения по payload.
JS сохраняет insertion order ключей, Go рандомизирует map — без нормативного порядка
конформанс-сравнение невозможно в принципе.

**Рекомендация (пакет из трёх правил):**

1. **Семантика `[*]`:** матчит только сегменты-неотрицательные-целые индексы (т.е. элементы
   массивов и плоские ключи вида `accounts[3]`). Нецелые «индексы» не матчатся. Это надо
   проговорить, потому что pre-flattened payload допускает произвольные ключи.
2. **Порядок перечисления:** индексы по возрастанию числом; несколько `[*]` — лексикографически
   по кортежу индексов, внешний старше («одометр»): `[0][0], [0][1], [1][0], …`
3. **Историческая формулировка порядка `issues[]`:** порядок шагов документа (DFS по flow,
   включая condition.steps и суб-пайплайны в точке вызова) → внутри одного правила порядок
   wildcard-элементов из п.2 → summary-issue strict-эскалации. D26 заменяет `flow`
   единым `steps`, D29 полностью удаляет strict-эскалацию; актуальный порядок задан SPEC.

**Цена:** нулевая для семантики (совпадает с фактическим Node-поведением на массивах),
но текущее поведение на «дырявых» flat-ключах (`accounts[5]` без `[0..4]`) надо проверить
и покрыть фикстурами.

## D6. Каноническая сериализация для `sourceHash` — APPROVED; вход хеша заменён D28

**Контекст.** `compileSnapshot()` верифицирует `sourceHash` → каждая реализация обязана уметь
*пересчитать* хэш, а не только сравнить строку. SHA-256 «over artifacts» без канонизации JSON
невоспроизводим: порядок ключей, представление чисел, escaping строк различаются между языками.

**Варианты:** (a) RFC 8785 (JCS); (b) собственная схема канонизации; (c) убрать пересчёт из
рантайма — хэш считает только билд-инструмент, реализации сравнивают строку.

**Рекомендация: (a) JCS.** Готовые реализации есть для JS/Java/Go; сериализация чисел в JCS
определена через ECMAScript `Number::toString`, что согласуется с D1 (binary64). Вариант (c)
ослабляет integrity-проверку снэпшота — отклонить.

**Цена:** нужно проверить, что текущий `computeSourceHash` совпадает с JCS; если нет —
пересчёт хэшей существующих снэпшотов при переходе на контракт (version bump снэпшота
это легализует).

## D7. Нормативная vs информативная поверхность результата — APPROVED; уточнено D28/D30

D30 удаляет из результата `control`, `kind`, trace и версию движка; D28 удаляет мету
снэпшота. Старый текст ниже сохранён как история решения.

**Контекст.** Не всё в результате можно и нужно требовать к побайтовому равенству.

**Рекомендация — двухуровневый контракт:**

**Нормативно (равенство обязательно):**
- `status`, `control`;
- `issues[]`: состав, порядок (D5), поля `kind, level, code, message, field, ruleId, pipelineId,
  stepId, expected, actual, meta` — `message` нормативен, т.к. это passthrough данных артефакта,
  а не текст реализации;
- ABORT: `error.code` и `error.details` нормативны; `error.message` — информативен
  (текст свободный, сравнивается только код);
- `ruleset.sourceHash`, `ruleset.projectId`, `ruleset.rulesetVersion`;
- вердикт валидации снэпшота/артефактов: конформные реализации обязаны **отвергать одно и то же
  множество входов** (сам факт reject нормативен).

**Информативно (структурный контракт без равенства значений):**
- `trace` целиком (поле `at` — timestamp, недетерминирован by design; состав шагов — SHOULD);
- `ruleset.engineVersion` (implementation-specific);
- тексты и коды диагностик валидации (`diagnostics[].code` — SHOULD совпадать, MUST — только
  сам вердикт ok/не-ok). Фиксировать все коды диагностик нормативно — тяжело и мало даёт;
  отложить до v2 контракта.

## D8. Представление отсутствующих значений в результате — APPROVED

**Контекст.** В JS `actual: undefined` молча не сериализуется. В Java/Go «undefined» нет —
надо выбрать между отсутствием ключа и `null`, и это видимое расхождение в JSON-результате.

**Рекомендация:** нормативный результат — чистый JSON; «нет значения» = **ключ отсутствует**
(не `null`). Единственное исключение — обязательный `field: null` у issue, которое нельзя
связать с одним основным полем. `actual` для отсутствующего поля payload → ключ `actual`
опущен.

## D9. Защитные reject'ы как часть контракта — APPROVED

**Контекст.** `__proto__`/`prototype`/`constructor`, глубина 256, циклы. Для Go/Java prototype
pollution не существует — соблазн реализации «не проверять». Но тогда один payload даёт
`ABORT DANGEROUS_PAYLOAD_KEY` на Node и `OK` на Go — контракт нарушен.

**Рекомендация:** все защитные reject'ы — **MUST для всех реализаций**, независимо от того,
являются ли они защитой на конкретной платформе. Формулировать не как «защита от prototype
pollution», а как «зарезервированные ключи, запрещённые контрактом». Коды ABORT
(`DANGEROUS_PAYLOAD_KEY`, `PAYLOAD_TOO_DEEP`, `ARTIFACT_TOO_DEEP`, `OPERATOR_FAULT`,
`OPERATOR_CONTRACT_VIOLATION`, `TRACE_REDACTOR_ERROR`, `RUNTIME_ABORT`) — исторический
перечень прототипа. Закрытый enum RC.5 содержит только коды из SPEC §6.7;
`ARTIFACT_TOO_DEEP`, `TRACE_REDACTOR_ERROR` и `RUNTIME_ABORT` в контракт не входят.

## D10. Кастомные операторы и неизвестные операторы — APPROVED; уточнено D26/D27

**Утверждённая модель: спека гарантирует только себя; кастомные операторы — специфика
реализации, вне спеки.**

Гарантия «один результат на любом рантайме» выдаётся строго в границах `jsonspecs/spec`
версии X: снэпшот, использующий только встроенные операторы спеки, обязан давать идентичный
нормативный результат на любой конформной реализации. Всё сверх этого — обещание конкретной
реализации в рамках её собственной документации и тестов, спека этим не занимается.

Что спека при этом определяет (интерфейс расширения, не более):
- форму результата любого оператора (`PASS|FAIL|SKIP`) и реакцию рантайма на её нарушение
  (`OPERATOR_CONTRACT_VIOLATION`, `OPERATOR_FAULT`);
- правило неизвестного оператора: снэпшот со ссылкой на оператор, отсутствующий в реестре
  реализации, **отвергается до любого исполнения**, нормативный код `OPERATOR_NOT_FOUND`.
  Обоснование детерминизма: при runtime-модели вердикт зависел бы от того, какой пайплайн
  запустили; reject-до-исполнения даёт один вердикт на снэпшот, без которого «множество
  отвергаемых входов» (D7) неопределимо;
- множество внешних операторов выводится из достижимых rule-артефактов; удалённое
  `requires.operators` не является полем fv2.

Автор реализации, встретивший чужой кастомный оператор, выбирает сам: реализовать
у себя или честно отдать `OPERATOR_NOT_FOUND`. Первоначальное решение отклоняло отдельный
межрантаймовый профиль пакета операторов как преждевременный. Конкретный план Node/Java
опроверг эту предпосылку; [DR-VIII] вводит отдельный профиль для пакетов, заявляющих
эквивалентность между рантаймами, не добавляя данные о пакете в снэпшот ядра.

## D11. Legacy-поверхности — APPROVED: резать через границу `formatVersion`

**Решение утверждено: контракт описывает только `formatVersion: 2`; legacy живёт в зоне
`formatVersion: 1`, которая целиком вне контракта.**

**Важное уточнение скоупа: контекст из контракта не уходит.** Нормативный вход исполнения —
кортеж `(prepared_snapshot, pipelineId?, payload, context?)`: контекст — явный отдельный
параметр, `$context.*` в правилах остаётся нормативной частью контракта. Поле
`required_context` позднее удалено D14. Здесь под нож идёт транспортный дубликат — legacy-канал
передачи контекста *внутри payload* под магическим ключом `__context`. Его удаление усиливает
свойство «движок = чистая функция»: payload становится только данными, без ключей, меняющих
семантику исполнения.

Логика минимального компромисса для рынка: каждая legacy-поверхность, включённая в контракт, —
это обязательный код в *каждой* новой реализации на *каждом* языке, навсегда. Ни Java-, ни
Go-, ни C#-реализация не имеют исторических пользователей `__context` — заставлять их нести
этот балласт ради совместимости, нужной только Node, и есть тот самый компромисс, которого
надо избежать. Граница `formatVersion` делает это чисто:

| Поверхность | В контракте (fv2) | Вне контракта (fv1, только Node) |
|---|---|---|
| `paths[]` в `any_filled` | нет — только `fields[]` | принимается |
| `payload.__context` | нет — `__context` теряет спецсемантику, это обычный ключ payload | принимается как источник контекста |
| Старая сигнатура `runPipeline(...)` | API вне скоупа контракта вообще (D12) | — |

Критичный нюанс по `__context`: это поверхность *рантайм-входа*, а не артефакта, поэтому
formatVersion напрямую её не гейтит. Решение: Node-реализация включает legacy-обработку
`__context` только для prepared-артефактов, собранных из fv1-снэпшотов; для fv2 поведение
строго контрактное. Иначе один и тот же payload с ключом `__context` даст разный результат
на Node и на Go — расхождение вердикта на ровном месте.

Миграция fv1 → fv2 — чисто механическая (`paths` → `fields`, `__context` → `context`
на стороне вызывающего сервиса), делается командой в `jsonspecs-cli`. Стоимость несут
существующие Node-потребители один раз, а не весь будущий рынок реализаций постоянно.

## D13. Семантика отсутствия поля: presence- vs value-операторы — APPROVED

**Утверждено (по инициативе владельца, обобщено на всё семейство операторов):**
обязательность поля — всегда отдельное правило аналитика; value-операторы никогда её
не подразумевают.

- **Presence-семантика** (отсутствие — часть предметной области оператора):
  `not_empty` (absent → FAIL), `is_empty` (absent → PASS), `not_true` (absent → PASS),
  `any_filled` (absent = пусто).
- **Value-семантика** (все остальные, включая `is_*`-типовые, сравнения, regex, словари,
  `field_*_field` по обоим операндам): при отсутствии поля оператор **не вызывается** —
  правило получает `SKIP`. В rule-шаге он ничего не порождает, в `when` становится
  `false`. Факт пропуска пишется в trace (информативная поверхность, D7).

**Обоснование (дизайн-принцип DSL, не равнение на отрасль):** обязательность поля —
явное решение аналитика, и ошибка *отсутствия* — отдельная диагностика от ошибки
*оператора*: она заслуживает собственного правила с собственными `code` и `message`,
которые выбирает аналитик, а не generic-ошибки, вшитой в каждый value-оператор.
Совпадение с моделью JSON Schema (`pattern`/`enum` отдельно от `required`) —
побочное, не мотивирующее. Аудит боевого пакета: 127 value-проверок, все 12 правил
на полях без парной проверки наличия уже обёрнуты в condition-гарды `*_if_present` —
вердикты пакета не меняются, а сами гарды становятся удаляемым boilerplate
(~12 условий + предикаты). Это отменяет прежнюю формулировку поправки D4 в части
«absent → FAIL» для `not_matches_regex`/`not_in_dictionary`: оба — value-семантика, skip.

## D14. `required_context` выпиливается из DSL — APPROVED

**Утверждено:** поле `required_context` пайплайна не существует в контракте (fv2;
закрытые схемы артефактов его отвергают). Это D13, применённый к контексту: полнота
runtime-контекста — явное решение аналитика, выраженное обычными правилами на путях
`$context.*` (например, `not_empty` на `$context.currentDate` с уровнем, `code` и
`message`, которые выбирает сам аналитик, обычно первым шагом entrypoint-флоу).
Выразительность растёт: `required_context` умел только «EXCEPTION и стоп», правило —
любой уровень.

Каскадные следствия: снимаются вопросы §5.5 (чей required_context проверять при
суб-пайплайнах — некого) и §5.8 (генерация кодов `CTX.<KEY>.REQUIRED` из имён ключей —
кодов нет, аналитик пишет свои). Важное взаимодействие с D13: value-проверки с absent
`$context.*`-операндом скипаются, поэтому сценарий, зависящий от контекста, ОБЯЗАН
гардить его явно, если отсутствие — ошибка.

Миграция боевого пакета: 6 entrypoint-пайплайнов с `required_context: ["currentDate"]`
→ одно общее правило `not_empty $context.currentDate` (EXCEPTION, авторский код)
первым шагом каждого из шести флоу. Потребители, матчившиеся на коды `CTX.*.REQUIRED`,
переходят на авторский код — входит в major-миграцию v3.

**[DR-II]** Попутно в Части II утверждены: закрытые схемы артефактов (§4.1), невалидность
rule-шага на предикат (§4.8), `specVersion` вместо `engine.minVersion` (§4.9),
`details` без `availablePipelines` (§5.1), inner-to-outer порядок вложенных
strict-summary (§5.7).

## [DR-III] Дополнение: утверждения Части III (result contract) — APPROVED

- **`expected` у `field_*_field`** = разрезолвленное значение `value_field` (Node 2.4.0
  его опускал). Следствие D13: issue возникает только при обоих присутствующих
  операндах, comparand всегда определён.
- **Групповые summary-issue**: рантаймовые факты — в объект `details` (не в `meta`,
  который целиком авторский; не под именем `aggregate` — имя называет роль, а не
  источник, и зеркалит `error.details`). Дискриминатор потребителя: `details.mode`.

- **`rule.issue.meta` пробрасывается в issues verbatim**; рантайм не пишет в него никогда

- **ABORT details**: `PAYLOAD_TOO_DEEP` → `{"maxDepth": 256}` без пути в нормативной
  поверхности (путь зависит от порядка обхода); `DANGEROUS_PAYLOAD_KEY` при нескольких
  запрещённых ключах репортит лексикографически наименьший путь (детерминизм details).
- **`ruleset.specVersion`** — нормативное эхо контракта поведения в результате;
  `engineVersion` — информативен.
- **Два канала отказа**: reject снэпшота (вердикт нормативен, форма — нет; единственный
  нормативный идентификатор — `OPERATOR_NOT_FOUND`) и ABORT исполнения (закрытый enum
  кодов). Конформанс — относительно равных наборов зарегистрированных операторов;
  для чисто встроенных снэпшотов — безусловен.

## [DR-I] Дополнение: утверждения Части I (data model / операторы) — APPROVED

- **Расширение D3 на строковое семейство** (`contains`, `matches_regex`,
  `not_matches_regex`): нестроковое значение → FAIL, хостовая стрингификация
  не происходит никогда. Мотив тот же, что в D3: прототип коэрсил через `String()`

- **`onEmpty`** унифицирован D20 до `PASS|FAIL|SKIP`; старые значения

- **Erratum rc.1: `onEmpty: "ERROR"` удалён.** Опция обещала ABORT, для которого в

  единственное место, где содержимое правила порождало бы анонимный аборт, ломая

  отсутствуют).

В спеке эти пункты помечены ссылкой `[DR-I]`; блоки Части II и Части III — `[DR-II]`

## D15. Нормативный вход — только вложенный JSON — APPROVED

**Контекст (внешнее ревью rc.1, блокер).** Двухформенный вход fv1 (вложенный ИЛИ
плоский payload) неоднозначен: `{"a.b": 1, "a": {"b": 2}}` — какое значение у пути
`a.b`? Определение экранирования, детекции режима и коллизий раздуло бы контракт.

**Утверждено:** нормативный вход fv2 — только обычный вложенный JSON; плоская карта —
внутренняя проекция, определяющая резолюцию путей; приём плоского входа — адаптер
конкретной реализации вне контракта. Ключи payload/context, ломающие адресацию —
пустой `""` и содержащие `.` `[` `]`, — отвергаются. Симметричные коды:
`INVALID_PAYLOAD`/`INVALID_CONTEXT` (не-объект, `details: {"expected":"object"}`),
`DANGEROUS_*_KEY`/`INVALID_*_KEY` (`details: {parentPath, key}`; parentPath от корня
контейнера, корень `""`), `PAYLOAD_TOO_DEEP`/`CONTEXT_TOO_DEEP`. Порядок проверок:
тип контейнера → скан ключей (top-down, в поддерево нарушения не спускаемся —
предки видимого нарушения всегда чисты, parentPath корректен; приоритет: опасные
payload → опасные context → недопустимые payload → недопустимые context; внутри
класса — лексикографический минимум пары `(parentPath, key)`) → глубина.

**Следствия:** разреженные wildcard-совпадения ОСТАЮТСЯ (дыра возникает от
отсутствия дочернего поля: `{"x":[{"v":1},{},{"v":2}]}` → `x[0].v`, `x[2].v`) —
правило порядка D5 не упрощается; все фикстуры с плоскими payload переписаны на
вложенные; интеграции с плоскими payload мигрируют через адаптер/CLI Node v3.

## D16. Линейность regex — свойство языка, не исполнения — APPROVED

**Контекст (внешнее ревью rc.1, блокер).** Формулировка «паттерны подмножества
исполняются за линейное время» неверна: грамматика допускает `(a+)+$`, и
бэктрекинговые движки (V8, java.util.regex) исполняют его суперлинейно.

**Утверждено:** нормативно — «язык реализуем автоматным движком за линейное время»
(подмножество RE2); плюс информативная security-нота (SHOULD автоматный движок или
эквивалентные митигации), без мандата на конкретную библиотеку [D12]. Для
Node-реализации v3 (роадмап, не спека): зависимость `re2` (npm-пакет называется
`re2`); её соответствие ДОКАЗЫВАЕТСЯ прогоном полного regex-набора фикстур, включая
folding (у re2 версия Unicode может отличаться от закреплённой — компенсация
обязательна), с пиннингом точной версии зависимости и банковской процедурой
поставки нативного модуля (зеркало npm, SBOM, проверенные бинарники/сборка из
исходников). Сервисный таймаут — только defense-in-depth: синхронный код прерывается
лишь worker'ом/отдельным процессом.

## D17. Граница конформанса кастомных операторов + контракт исхода — APPROVED

**Уточняет D10 и §7.1, в редакции D19.** Равные имена операторов не означают
равную семантику: межреализационное равенство результата гарантируется только для
снэпшотов исключительно на встроенных операторах. Общий контракт расширения —
строго `PASS|FAIL|SKIP`. Возвращённый `EXCEPTION` является значением вне enum и
даёт `ABORT OPERATOR_CONTRACT_VIOLATION`; брошенное исключение даёт
`ABORT OPERATOR_FAULT`. Реакция сайт-независима. Бизнес-уровень `EXCEPTION`
принадлежит `rule.issue`, а не оператору. Переносимая тестируемость обеспечивается
`conformance.rule.throw`, `conformance.rule.invalid_result` и
`conformance.rule.tri`, регистрируемыми только conformance-прогонщиком.

## D18. Пиннинг Unicode для case folding — заменено D29

Решение относилось к удалённому regex-флагу `i` и сохранено только как объяснение
стоимости этого флага для нескольких рантаймов.

Флаг `i` — Unicode simple case folding по **Unicode 16.0.0**, нормативная ссылка на
неизменяемый файл <https://www.unicode.org/Public/16.0.0/ucd/CaseFolding.txt>,
только статусы `C` и `S` (`F` и `T` не используются). Эквивалентность симметрична и
транзитивна: точки эквивалентны ⟺ их фолдинги равны (не односторонняя замена).
Реперные факты для фикстур: `и`≡`И`, `K`(U+212A)≡`k`, `ſ`≡`s`, `Σ`≡`σ`≡`ς`,
`ß`≡`ẞ`(U+1E9E), `ß`≢`SS` (full-фолдинг исключён), турецкие `İ`/`ı` фолдятся сами в
себя и не матчат ASCII `i`/`I`. Хэш таблицы не требуется: версии + неизменяемой
ссылки достаточно. Реализация, чей движок несёт иную версию Unicode, MUST
компенсировать; соответствие подтверждается исполнением фикстур, а не
предположением о совместимости.

## D19. Единое правило без ролей check/predicate — APPROVED

**Утверждено для rc.3.** Поле `role` удаляется. Оператор всегда возвращает один
сайт-независимый исход `PASS|FAIL|SKIP`; правило становится одним переиспользуемым
бизнес-условием. Rule-шаг интерпретирует `FAIL` как issue, а `when` интерпретирует
`PASS` как `true`, `FAIL`/`SKIP` как `false` и никогда не создаёт issue.

Диагностическая декларация — опциональный цельный объект
`issue: {level, code, message, meta?}`. Он не влияет на логику. Поэтому правило с
`issue.level: EXCEPTION` разрешено в `when`, а правило без `issue` разрешено как
условие; только rule-шаг на правило без `issue` отвергается. `meta` перемещается
внутрь `issue`, поскольку является данными создаваемого issue. Уникальность кода
проверяется среди правил с `issue`.

Возвращаемый оператором `EXCEPTION` удалён: динамическая бизнес-эскалация нарушала
владение уровнями аналитиком. Значение вне `PASS|FAIL|SKIP`, включая `EXCEPTION`,
даёт `OPERATOR_CONTRACT_VIOLATION`; брошенный технический сбой — `OPERATOR_FAULT`.
Это заменяет ролевую часть D10/D13/D17 и удаляет отдельные таблицы предикатов.

## D20. Единая wildcard-агрегация и эффективная популяция — APPROVED; MIN/MAX удалены D29

**Утверждено для rc.3.** Логический `mode` отделён от формы репортинга:
`mode = ALL|ANY|COUNT|MIN|MAX`; `issueMode = EACH|SUMMARY` существует только для
`ALL`/`ANY`. При наличии `issue` он обязателен для этих двух режимов; без `issue`
запрещён. `COUNT`/`MIN`/`MAX` всегда дают summary при провале. Legacy `EACH` как
логический mode и `summaryIssue` удалены.

Wildcard требует явного `aggregate`; без wildcard он запрещён. `value_field` не
содержит wildcard. `onEmpty = PASS|FAIL|SKIP`, дефолт `SKIP`, и относится только к
нулю структурных совпадений. `SKIP`-элементы исключаются из эффективной популяции;
если структурные элементы были, но все дали `SKIP`, итог — `SKIP` независимо от
`onEmpty`. Это сохраняет D13: неприменимость не превращается в ошибку наличия.

Групповые details различают `matched`, `evaluated`, `skipped`, `passed`, `failed`.
Изменение COUNT признано семантическим: ранее скипнутые check-элементы могли
засчитываться прошедшими; теперь они не участвуют. Смешанные популяции переносимо
проверяет `conformance.rule.tri`.

## [DR-V] Дополнение: унификация rc.2 → rc.3 — APPROVED

rc.3 планировался последним архитектурным редизайном до `1.0.0`. Проверка боевого
корпуса бенефициаров и требований хранения в БД выявила оставшееся смешение
authoring/runtime-моделей и неоднозначности семантики. Это процессное решение
заменяется явно утверждённым пакетом rc.4 D21–D25 ниже. Скалярные
`greater_or_equal`/`less_or_equal` остаются кандидатами бэклога без номера решения.

## D21. Точные id, явные exports и замкнутый исполняемый бандл — APPROVED

**Утверждено для rc.4.** Id артефактов — непрозрачные глобально уникальные строки.
Каждая ссылка — точный id; скоупы, видимость, относительное раскрытие, сиротские
скоупы, longest-prefix-владение и особая семантика `library.*` удаляются. Импорты,
алиасы, папки и таксономия принадлежат authoring/build-инструментам.

`description` удаляется из артефакта как неисполняемая метаинформация. Флаг
`pipeline.entrypoint` и выбор пайплайна по умолчанию удаляются; каждый вызов передаёт
`pipelineId`. В снэпшоте появляется обязательный непустой `exports` со списком id пайплайнов: это
публичный API бандла и корни достижимости, а не механизм дефолтного выбора.
Отсутствующий/неверно типизированный id даёт `INVALID_PIPELINE_ID`, неизвестный или
неэкспортированный — `PIPELINE_NOT_FOUND`.

Транзитивное замыкание от exports MUST в точности совпадать с множеством артефактов.
Исходный проект может хранить неиспользуемые файлы, production-снэпшот — нет. Замыкание
идёт по шагам, rule-листьям `condition.when` и нормативным ссылкам на словари.
Управляющий DAG исправляется: он включает pipeline и condition, закрывая латентные
циклы condition→condition и pipeline→condition→pipeline.

Отдельный condition-артефакт осознанно сохраняется. В боевом корпусе 96 conditions,
40 из них вызываются более чем из одного места, а цепочки достигают pipeline плюс трёх
вложенных condition. Встраивание упростило бы runtime-схему ценой глубокой JSON-лапши,
что противоречит назначению DSL.

## D22. Детерминированный short-circuit гардов — APPROVED; экстремумы удалены D29

**Утверждено для rc.4.** `when` вычисляется рекурсивно, слева направо, с обязательным
short-circuit: `all` останавливается на первом false, `any` — на первом true. Это
наблюдаемо, когда следующий пользовательский оператор бросает исключение, поэтому
полный обход неконформен. Фикстуры закрепляют `any(PASS, throw)`,
`all(FAIL, throw)` и противоположные случаи без ранней остановки.

`MIN`/`MAX` выбирают исходный структурный экстремум до вычисления оператора и затем
вызывают оператор ровно один раз. Неклассифицируемое значение или смешение видов даёт
FAIL до вызова; равенство разрешается первым элементом нормативного порядка; исход
выбранного элемента, включая SKIP, становится исходом агрегата. Эффективная популяция
остаётся только у `ALL`/`ANY`/`COUNT`.

## D23. Замкнутая область конечных binary64-чисел — APPROVED

**Утверждено для rc.4.** Каждый числовой токен JSON преобразуется в IEEE 754 binary64
round-to-nearest, ties-to-even. Любой конечный результат принимается; переполнение до
бесконечности отвергается, underflow следует binary64, отрицательный ноль нормализуется
в положительный. Прежняя оговорка «вне гарантии детерминизма» удаляется: `0.1`
принимается, а `9007199254740993` нормативно округляется до `9007199254740992`.
Переполнение payload/context даёт `INVALID_PAYLOAD_NUMBER`/
`INVALID_CONTEXT_NUMBER` с лексикографически минимальным путём. Строка считается
числовой для упорядоченного сравнения только при конечном результате преобразования;
`"1e400"` — валидная строка, но неклассифицируема для порядка.

## D24. Закрытые вложенные схемы и `params` пользовательского оператора — APPROVED; уточнено D26-D28

**Утверждено для rc.4.** Конверт снэпшота, `requires`, `exports`, `aggregate`, ссылка
на статический словарь, объектная запись словаря, объекты `when` и шаги явно закрыты.
В проекте rc.4 `snapshot.meta` сохранялся; D28 удаляет его. D26 удаляет `requires` и объектные
ссылки на словари; D27 добавляет именованные разрешаемые `inputs`.

Специфичные оператору верхнеуровневые поля rule запрещены. Пользовательское правило
использует единый JSON-объект `params`, а зарегистрированный оператор предоставляет
закрытую compile-time-схему и список принимаемых стандартных операндных полей. Ядро
отвергает несоответствие до исполнения и передаёт принятый params дословно. API
регистрации и язык схемы — детали реализации, но межрантаймовый пакет обязан
публиковать эквивалентные машиночитаемые схемы. Строки params не создают рёбер графа;
зависимость от словаря выражается нормативным `dictionary`.

## D25. Независимый от порядка хэш исполняемого бандла — полностью заменено D28

**Утверждено для rc.4; заменяет artifacts-only-проекцию D6.** `sourceHash` — SHA-256
от JCS `{requires, exports, artifacts}` после сортировки имён операторов,
экспортированных pipeline id и артефактов по id. Отсутствующий `requires` проецируется
в пустой список операторов. Вложенные семантические массивы (`flow`, `steps`, `when`,
entries словаря) сохраняют порядок. Порядок строк БД перестаёт влиять на хэш, а
публичный API и требования операторов защищаются тем же идентификатором целостности,
что и исполняемый граф. `meta` исключён.

## [DR-VI] Дополнение: упрощение исполняемого бандла rc.3 → rc.4 — APPROVED

rc.4 — финальный архитектурный кандидат до 1.0. Release-gate MUST закрепить новое
замыкание, объединённый DAG, short-circuit, экстремумы, binary64, закрытые вложенные
схемы, params-схемы и хэш положительными и rejection-фикстурами и быть показательно
красным на намеренном нарушении до тега. Дальнейшие изменения до `1.0.0` ограничены
errata, найденными при реализации Node и Java именно против этого кандидата.

## D26. Артефакты-карта и одна форма шага — APPROVED

**Утверждено для rc.5; заменяет array/id и типизированные шаги из D11, D21 и D25.**
`artifacts` — объект, ключом которого служит глобально уникальный id; внутри артефакта
поле `id` не повторяется. Pipeline и condition используют одинаковое поле `steps`, а
каждый шаг — точная строка id. Тип цели определяется самим артефактом. `stepId` удалён.

`requires.operators` удалён: множество внешних операторов выводится из достижимых
правил как использованные имена минус встроенные. Отсутствующий оператор по-прежнему
даёт нормативное отвержение `OPERATOR_NOT_FOUND`. Ссылка на словарь стала строкой id,
а записи словаря — уникальными скалярами.

## D27. Разрешаемые ядром входы оператора и константные params — APPROVED

**Утверждено для rc.5; уточняет D10, D17, D19 и D24.** Каждый оператор регистрирует
закрытый compile-time-контракт: стандартные операнды, именованные `inputs` и
константные `params`. `inputs` сопоставляет объявленные оператором имена скалярным
путям без wildcard. Ядро проверяет и разрешает пути до вызова и не передаёт оператору
payload, context, пути либо resolver API.

Во время вызова отсутствующий путь означает отсутствие соответствующего ключа, а
JSON `null` — присутствующий ключ со значением `null`. Точная абстрактная запись
вызова закреплена в SPEC §3.1. Результат оператора остаётся ровно
`PASS|FAIL|SKIP`; throw/panic даёт `OPERATOR_FAULT`, а не четвёртый исход. Для issue
пользовательского оператора `actual` берётся из основного `field`, `expected` — из
`value` либо разрешённого `value_field`; отсутствующие значения не добавляются.

Errata RC.5 уточняет: обязательное имя `inputs` обязательно в конфигурации rule, но путь
может не разрешиться во входе. В таком случае ключ отсутствует в карте `inputs`, оператор
всё равно вызывается. Автоматический `SKIP` применяется к любому настроенному стандартному операнду
`field`/`value_field` пользовательского оператора, независимо от его обязательности в контракте; наблюдение отсутствия
выражается только через `inputs`. Ограничения типов на этапе компиляции описывают авторскую конфигурацию, а не
значения, позже разрешённые из путей. `fields` разрешён только встроенному `any_filled`; `value` и `value_field` взаимоисключающие.
Реестр связывает каждое непустое имя ровно один раз, а пакеты пользовательских операторов не могут затенять имена встроенных.

## D28. I-JSON, хэш всего снэпшота и удаление snapshot meta — APPROVED

**Утверждено для rc.5; заменяет D6 и D25.** Текстовый адаптер до обычного lossy parse
отвергает повторяющиеся имена членов объекта и одиночные суррогаты. JCS сортирует по
беззнаковым UTF-16 code units как RFC 8785; поэтому U+10000 идёт раньше U+E000.

`sourceHash` равен SHA-256 от JCS всего снэпшота, из которого удалено только поле
`sourceHash`. `exports` должен уже прийти уникальным и строго отсортированным; проверка
не нормализует его. `snapshot.meta` удалён: иначе одинаковый hash мог давать разные
результаты. Авторский `rule.issue.meta` сохранён, входит в hash и нормативно
пробрасывается в issue.

## D29. Удаление неподтверждённых возможностей версии 1 — APPROVED

**Утверждено для rc.5; заменяет соответствующие части D4, D18, D20 и D22.** Из версии
1 удалены pipeline `strict`, агрегаты `MIN`/`MAX` и regex-флаги. В исследованных
боевых пакетах они не используются, но требуют непропорционально сложной
межрантаймовой семантики. Они могут вернуться позднее с реальным сценарием и
переносимыми golden-векторами. `ALL`, `ANY` и `COUNT` сохранены.

Regex получает ровно строку паттерна после декодирования I-JSON; дополнительного
схлопывания обратных косых черт нет. Сравнения остаются отдельными читаемыми
операторами DSL; реализация может свести их к одному внутреннему механизму.

## D30. Минимальный нормативный результат — APPROVED

**Утверждено для rc.5; уточняет D7 и D8.** `control` удалён, потому что полностью
выводится из `status`; постоянный `kind` удалён; `stepId` исчез вместе с объектными
шагами. Trace и версия движка относятся к API реализации, а не к результату ядра.
`ruleset` содержит только нормативные `specVersion` и `sourceHash`.

## [DR-VII] Дополнение: финальный пересмотр формата rc.4 → rc.5 — APPROVED

rc.5 — сознательный последний пересмотр формата, а не полировка rc.4. Release-gate
MUST покрыть карту артефактов, строковые шаги, вывод внешних операторов, разрешение
`inputs` ядром, различие absent/null, I-JSON до parse, UTF-16-порядок JCS между Node и
Java, хэш всего снэпшота, удалённые legacy-поля и сохранённый хэшируемый
`rule.issue.meta`. После rc.5 до 1.0 допускаются только errata.

## [DR-VIII] Дополнение: семантическое замыкание RC.5 — APPROVED

Ревью реализации RC.5 выявило требования, которые были задуманы, но оставались
неоднозначными или неисполняемыми. Редакция RC.5 закрепляет без изменения формата:

- точные схемы операндов пользовательских и conformance-операторов;
- вызов операторов при отсутствующих именованных `inputs`;
- числовые строки в скалярных сравнениях порядка;
- только экранированный литеральный дефис, абсолютные regex-якоря и dot, исключающий
  только LF;
- полный последовательный обход агрегатов, атомарное создание `EACH`-issue и форму
  `COUNT + onEmpty`;
- технический `ABORT` отбрасывает ранее накопленные бизнес-issue, а `EXCEPTION` их сохраняет;
- `OPERATOR_NOT_FOUND` только после всех успешных проверок, не требующих недоступного контракта
  отсутствующего оператора;
- закрытые объекты результата, issue, error и ruleset;
- отдельный версионируемый межрантаймовый профиль пакета операторов с диапазоном версий ядра,
  общими эталонными векторами и сведениями о сборке с указанием алгоритма контрольной суммы.

Скалярные `greater_or_equal`/`less_or_equal` остаются неутверждённым бэклогом и в эту
редакцию не входят.
